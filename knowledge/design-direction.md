# デザインディレクション定義

## 概要

このプロジェクトで生成する UI のデザイン方向性・判断基準を定義する。
トンマナ選定やコンポーネント設計の上位指針として機能する。

## ターゲット UI

SaaS ダッシュボード（業務管理・データ可視化）とコーポレートサイト（LP・ブランドサイト）の両方を対象とする。案件ごとに適用するパターンを選択する。

## デザイントーンの候補

以下の3候補から、プロジェクトの方向性を選定する。

### 候補A: モダン・ミニマル

| 項目 | 内容 |
|------|------|
| キーワード | クリーン、余白、静謐、フラット、呼吸感 |
| 印象 | 洗練された、無駄のない、プロダクト感 |
| 適性 | SaaS ダッシュボード向き。情報密度が高い画面でも読みやすい |
| 参考サイト | Stripe（stripe.com）、Linear（linear.app）、Vercel（vercel.com） |

特徴:
- 余白を多く取り、要素の密度を下げる
- モノクロ + アクセントカラー1色の構成
- シャドウは控えめ、ボーダーで区切る
- アイコンは線画（outlined）

### 候補B: プロフェッショナル・信頼感

| 項目 | 内容 |
|------|------|
| キーワード | 堅実、信頼、安定、知的、明快 |
| 印象 | 安心感がある、ビジネス向け、落ち着いた |
| 適性 | コーポレートサイト・SaaS 両方。BtoB 寄りのプロダクトに強い |
| 参考サイト | Notion（notion.so）、GitHub（github.com）、Slack（slack.com） |

特徴:
- ニュートラルカラーをベースに、ブランドカラーを差し色に使う
- 情報のヒエラルキーが明確（見出し・本文・補助のサイズ差が大きい）
- カードベースのレイアウト
- アイコンは塗り（filled）またはデュオトーン

### 候補C: クリエイティブ・モダン（ハイブリッド）

| 項目 | 内容 |
|------|------|
| キーワード | 洗練、遊び心、ダイナミック、先進的、表現力 |
| 印象 | デザイン会社らしい、クリエイティブだが実用的 |
| 適性 | 自社ブランディングを兼ねる場合。SaaS・LP 両方で映える |
| 参考サイト | Figma（figma.com）、Framer（framer.com）、Arc（arc.net） |

特徴:
- グラデーションやブランドカラーを大胆に使う
- マイクロインタラクション（ホバー、トランジション）を重視
- ダークモード対応前提
- アイコンはカスタムまたはアニメーション付き

## アンチパターン

以下のスタイル・パターンは避ける。

### ビジュアル

| アンチパターン | 理由 |
|--------------|------|
| ネオンカラーの多用 | 可読性が下がり、長時間利用で疲労する |
| 過剰なシャドウ・グロー | 古く見える。2020年代のトレンドはフラット寄り |
| 装飾的すぎるフォント（筆記体・ディスプレイ体） | ビジネス用途で可読性が低い |
| 背景画像の全面使用 | テキストの視認性が下がる。ヒーローセクション以外では避ける |
| 原色（純粋な赤・青・緑）の直接使用 | 彩度が高すぎてハーシュ。トーンを調整したカラーを使う |

### レイアウト

| アンチパターン | 理由 |
|--------------|------|
| 要素の詰め込みすぎ | 情報過多で認知負荷が高い。余白を確保する |
| 一貫性のないスペーシング | 雑に見える。8px グリッドに統一する |
| 横スクロール（テーブル以外） | モバイルで破綻する。レスポンシブを前提に設計する |
| 3カラム以上のダッシュボード | 情報密度が高くなりすぎる。2カラム（サイドバー + メイン）が基本 |

### インタラクション

| アンチパターン | 理由 |
|--------------|------|
| 派手すぎるアニメーション | 操作の邪魔になる。トランジションは 200-300ms が適切 |
| 確認ダイアログの乱用 | ユーザーが慣れて読まなくなる。Undo 可能な設計を優先する |
| ツールチップ依存の UI | 重要な情報がホバーしないと見えない。ラベルを直接表示する |
| 自動カルーセル | ユーザーが読んでいる途中で切り替わる。手動操作のみにする |

### アクセシビリティ

| アンチパターン | 理由 |
|--------------|------|
| コントラスト比 4.5:1 未満のテキスト | WCAG AA 非準拠。視認性が低い |
| 色だけで状態を区別 | 色覚特性のあるユーザーが判別できない。アイコン・テキストを併用する |
| フォーカスインジケーターの非表示 | キーボード操作ユーザーが現在位置を見失う |
| フォントサイズ 12px 未満 | モバイル・高齢者ユーザーに読みにくい。最小 14px を推奨 |

## 設計原則

1. **Content First** — コンテンツが主役。装飾はコンテンツを引き立てるために使う
2. **Consistency** — 同じ意味には同じ見た目。コンポーネントとトークンで一貫性を保つ
3. **Progressive Disclosure** — 情報は段階的に開示する。最初から全てを見せない
4. **Accessible by Default** — アクセシビリティは後付けではなく最初から組み込む
5. **Responsive First** — モバイルから設計し、デスクトップに拡張する

## デザイン作成時のプロセス

案件ごとにデザインを作成する際、以下のプロセスを必ず実行する。

### 1. UI トレンド調査

デザイン作業の最初に、最新の UI/UX トレンドを Web 検索で調査する。

調査項目:
- 現在の UI デザイントレンド（レイアウト、カラー、タイポグラフィ、インタラクション）
- 案件の業界・業種に特化したデザイントレンド
- 競合・類似サービスの UI 傾向

調査結果は案件ごとの成果物フォルダに記録し、以降のステップの判断根拠とする。

### 2. トンマナ選定

上記のトレンド調査結果 + 本ドキュメントのデザイントーン候補を参考に、案件に適したトンマナをユーザーに提案する。

選定する要素:
- デザイントーン（候補 A/B/C またはカスタム）
- カラーパレット（WCAG AA 準拠のコントラスト比検証を含む）
- タイポグラフィ（フォントファミリー、モジュラースケール）
- スペーシング（グリッドシステム、ブレークポイント）

### 3. 参考サイト・パターン選定

本ドキュメントの「参考サイト集」「機能別 UI パターンリファレンス」から、案件に適したパターンを選定する。トレンド調査で見つけた新しい参考サイトがあれば追加する。

### 4. デザイン実行

選定したトンマナ・パターンに基づいてデザインを作成する。本ドキュメントの「アンチパターン」に該当しないことを確認しながら進める。
