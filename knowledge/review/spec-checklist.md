# 仕様書レビューチェックリスト

仕様書（`docs/specs/` 配下）をレビューする際に確認する観点です。

## 必須チェック項目

### 課題と解決策
- [ ] 課題（背景）が明確に書かれているか
- [ ] 解決策が課題に対して適切か（手段と目的がズレていないか）
- [ ] 他に良い解決策がないか検討されているか

### 工数・コスト
- [ ] 工数の見積もりが現実的か（S/M/L/XL の判定が妥当か）
- [ ] 得られる効果に対してコストが見合うか
- [ ] 隠れた工数（調査・調整・テスト）が漏れていないか

### ビジネスルール
- [ ] `knowledge/business-rules.md` のルール・制約に違反していないか
- [ ] セキュリティ上の問題がないか
- [ ] 法的・契約上の制約に抵触しないか

### 網羅性
- [ ] 影響を受けるステークホルダーが洗い出されているか
- [ ] リスクの洗い出しが十分か
- [ ] 依存関係（他の仕様・外部サービス）が明記されているか
- [ ] 成功基準（何をもって完了とするか）が定義されているか

### 実現可能性
- [ ] 現在の技術スタック・環境で実現可能か
- [ ] 必要な権限・アクセスが確保できるか
- [ ] スケジュール的に無理がないか

## 判定基準

| 判定 | 意味 | 次のアクション |
|------|------|--------------|
| ✅ 承認 | すべての必須項目をクリア | 実行フェーズに進んでOK |
| ⚠️ 条件付き承認 | 軽微な修正が必要 | 指摘箇所を修正してから実行 |
| ❌ 差し戻し | 重大な問題あり | 仕様書を修正して再レビュー |
