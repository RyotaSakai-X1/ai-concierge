# 並列実行ルール

## 概要

Agent tool の `isolation: "worktree"` を使用して、複数のイシューを並列で実装する際のルール。

## Worktree 分離ルール

### 基本原則

- 各サブエージェントは独立した git worktree で作業する
- worktree 間でファイルを直接共有しない
- 各 worktree は独自のブランチを持つ

### やってよいこと

- worktree 内でのファイル作成・編集・削除
- worktree 内でのコミット・プッシュ
- `gh` CLI によるPR作成
- `gh issue view` による情報取得

### やってはいけないこと

- 他の worktree のファイルを直接参照・変更
- main ブランチへの直接コミット
- force push
- 他のブランチの変更を取り込む（リベース等）

## コンフリクト検出アルゴリズム

### 事前検出（実行前）

イシューの本文から影響ファイルを推定し、重複をチェック:

1. イシュー本文の「やること」セクションからファイルパスを抽出
2. ファイルパスが重複するイシューペアを検出
3. 重複がある場合は警告を出す（実行は可能）

### 事後検出（実行後）

各ブランチの変更ファイルを比較:

```bash
# ブランチAの変更ファイル
git diff --name-only main..branch-a > /tmp/files-a.txt

# ブランチBの変更ファイル
git diff --name-only main..branch-b > /tmp/files-b.txt

# 重複チェック
comm -12 <(sort /tmp/files-a.txt) <(sort /tmp/files-b.txt)
```

重複ファイルがある場合:
- 警告を出し、マージ順序を提案する
- 先にマージする PR を推奨（変更行数が少ない方を先に）
- 後続 PR はリベースが必要になる可能性を報告

## マージ戦略

### 独立した変更（コンフリクトなし）

- 任意の順序でマージ可能
- 全て squash merge を推奨

### ファイル競合がある場合

1. 変更行数が少ない PR を先にマージ
2. 後続 PR のブランチで `git rebase main` を実行
3. コンフリクトがあれば手動解消
4. 解消後にマージ

### 論理的依存がある場合

1. 依存元の PR を先にマージ
2. 依存先の PR のブランチで `git rebase main` を実行
3. マージ

## エラーハンドリング

### サブエージェントが失敗した場合

- 他のサブエージェントは影響を受けず、そのまま続行する
- 失敗したサブエージェントのエラー内容を収集し、最終レポートに含める
- 失敗した worktree は変更なしとして自動クリーンアップされる

### 部分的成功の場合

- 成功した PR は通常通りマージ可能
- 失敗したイシューは手動対応または再実行を提案

### ネットワークエラーの場合

- `git push` や `gh pr create` のネットワークエラーは最大3回リトライ
- リトライ間隔: 2秒、4秒、8秒（指数バックオフ）
