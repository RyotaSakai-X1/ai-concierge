# 並列実行ルール（Agent Teams + worktree ハイブリッド）

## 概要

Agent Teams（メンバー間通信）と git worktree（ファイル分離）を組み合わせて、複数のイシューを並列で実装する。各チームメンバーは独立した Claude Code インスタンスとして動作し、共有タスクリスト経由で通信する。

## Agent Teams チーム分離ルール

### 基本原則

- 各チームメンバーは独立した Claude Code インスタンス + 独自 worktree で作業する
- メンバー間の通信は共有タスクリスト経由で行う（ファイルの直接共有はしない）
- 各メンバーは自分の worktree 内でのみファイル操作を行う
- テックリード（親エージェント）がチームの作成・監視・結果集約を担当する

### やってよいこと

- `git worktree add` による作業用 worktree の作成
- worktree 内でのファイル作成・編集・削除
- worktree 内でのコミット・プッシュ
- `gh` CLI による PR 作成
- `gh issue view` による情報取得
- 共有タスクリスト経由での他メンバーとの通信（ファイル宣言、進捗報告等）

### やってはいけないこと

- 他のメンバーの worktree のファイルを直接参照・変更
- main ブランチへの直接コミット
- force push
- 他のブランチの変更を取り込む（リベース等）

## メンバー間通信プロトコル

### ファイルロック宣言

各メンバーは実装開始前に、変更予定のファイル一覧を共有タスクリストに宣言する:

1. イシュー本文の「やること」から影響ファイルを特定する
2. 共有タスクリストに「メンバー#{番号}: {ファイルパス一覧}」を登録する
3. 他メンバーの宣言と重複がないか確認する

### 競合検出通知

ファイル宣言が重複した場合:

1. 重複を検出したメンバーが共有タスクリストに警告を登録する
2. テックリードが競合解消の調整を行う
3. 調整方法: ファイルの担当分担を決める / 一方を先に実行して他方を待機させる

### 進捗・結果の共有

- 実装完了時: PR URL と変更ファイル一覧を共有タスクリストに報告する
- エラー発生時: エラー内容を共有タスクリストに報告する

## コンフリクト検出アルゴリズム

### 事前検出（実行前）

イシューの本文から影響ファイルを推定し、重複をチェック:

1. イシュー本文の「やること」セクションからファイルパスを抽出
2. ファイルパスが重複するイシューペアを検出
3. 重複がある場合は警告を出す（実行は可能）

### 動的検出（実行中 — メンバー間通信）

メンバーのファイルロック宣言を使ったリアルタイム競合検出:

1. 各メンバーが実装開始前にファイル宣言を共有タスクリストに登録
2. 宣言が重複した場合、テックリードが調整指示を出す
3. 調整完了まで重複ファイルの編集を保留する

### 事後検出（実行後）

各ブランチの変更ファイルを比較:

1. `git diff --name-only main..branch-a` と `git diff --name-only main..branch-b` で変更ファイル一覧を取得
2. 重複するファイルがあればコンフリクトの可能性を警告
3. 重複ファイルがある場合はマージ順序を提案する（変更行数が少ない方を先に）
4. 後続 PR はリベースが必要になる可能性を報告

## マージ戦略

### 独立した変更（コンフリクトなし）

- 任意の順序でマージ可能
- 全て squash merge を推奨

### ファイル競合がある場合

1. 変更行数が少ない PR を先にマージ
2. 後続 PR のブランチで `git rebase main` を実行
3. コンフリクトがあれば手動解消
4. 解消後にマージ

### 論理的依存がある場合

1. 依存元の PR を先にマージ
2. 依存先の PR のブランチで `git rebase main` を実行
3. マージ

## エラーハンドリング

### チームメンバーが失敗した場合

- 他のチームメンバーは影響を受けず、そのまま続行する
- 失敗したメンバーのエラー内容を共有タスクリスト経由で収集し、最終レポートに含める
- 失敗したメンバーの worktree は変更なしとして自動クリーンアップされる

### 部分的成功の場合

- 成功した PR は通常通りマージ可能
- 失敗したイシューは手動対応または再実行を提案

### ネットワークエラーの場合

- `git push` や `gh pr create` のネットワークエラーは最大3回リトライ
- リトライ間隔: 2秒、4秒、8秒（指数バックオフ）

## フォールバック戦略

Agent Teams が未有効（`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` が未設定）の場合:

- 従来の Agent tool + `isolation: "worktree"` 方式で並列実行する
- メンバー間通信は利用不可。事前・事後のコンフリクト検出のみで対応する
- `/parallel-work` コマンドが自動的にフォールバックを判定する
