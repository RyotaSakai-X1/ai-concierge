# /estimating — 工数見積コマンド

機能要件が記載されたスプレッドシートに対して工数見積を人日単位で行い、結果をスプレッドシートに書き出す。

## 引数

- `$ARGUMENTS` — スプレッドシートの URL または ID

## Step 1: スプレッドシートの取得

入力: `$ARGUMENTS` からスプレッドシート ID を抽出（URL の場合は `/spreadsheets/d/` と `/edit` の間）

取得手順: `mcp__google-workspace__get_spreadsheet_info` → `mcp__google-workspace__read_sheet_values`（機能要件シートを優先）

gid パラメータがあれば対象シートの特定に使用。

## Step 2: 工数見積

入力: 各行の機能要件（機能名・カテゴリ・要件の詳細度）

### 機能タイプ別の基準工数

基本的に機能要件は API ＋ UI。

| タイプ | 内容 | 基準工数 |
|--------|------|----------|
| 標準的な UI | CRUD 画面、一覧/詳細画面 | 1〜2 人日 |
| 複雑な UI | ダッシュボード、複雑なフォーム、インタラクティブ機能 | 3〜5 人日 |
| 標準的な API | 複数テーブル結合、バリデーション | 2〜3 人日 |
| 複雑な API | 複雑なビジネスロジック、外部連携 | 4〜5 人日 |
| バッチ処理 | 定期実行、データ処理 | 2〜5 人日 |
| 認証/認可 | ログイン、権限管理 | 2〜5 人日 |
| 外部連携 | API 連携、ファイル取込 | 2〜10 人日 |

### 複雑度の調整係数

| 複雑度 | 条件 | 係数 |
|--------|------|------|
| シンプル | 明確な要件、既存パターンあり | ×0.8 |
| 標準 | 一般的な要件 | ×1.0 |
| 複雑 | 曖昧な要件、新規技術 | ×1.5 |
| 非常に複雑 | 未知の領域、高リスク | ×2.0 |

### 計算ルール

| ルール | 値 |
|--------|-----|
| バッファ | 素の工数 × 1.2（+20%） |
| 端数処理 | 小数点第 1 位まで（第 2 位を四捨五入） |
| 最小単位 | 0.5 人日 |

## Step 3: 見積結果のフォーマット

出力構成:

| セクション | 内容 |
|-----------|------|
| サマリー | 総機能数 / 見積工数（素） / バッファ込みをテーブルで |
| 詳細見積 | カラム: `#` / `カテゴリ` / `機能名` / `タイプ` / `複雑度` / `工数（人日）` / `バッファ込み` / `備考` |
| カテゴリ別集計 | カラム: `カテゴリ` / `機能数` / `工数（人日）` / `バッファ込み`（最終行に **合計**） |

## Step 4: スプレッドシートへの書き出し（必須）

工数列の特定: ヘッダー行から `工数` / `見積` / `人日` / `工数（人日）` を検索。列が存在しない場合は最終列の右隣に「工数（人日）」を作成。

`mcp__google-workspace__modify_sheet_values` でデータ行のみ書き出す（ヘッダー行は除く）。

書き出し完了後の報告項目: 書き出し先（シート名と列範囲） / 書き出し件数 / 工数合計

## 制約

- 要件が曖昧な場合は前提条件を明記し幅を持たせる（例: 1〜2 人日）
- 依存関係がある場合は先行機能の完了を前提とした旨を備考に記載
- 技術調査が必要な場合は調査工数を別途計上することを提案
- テスト工数は単体テストを含む。結合テスト・システムテストは別途確認
- スプレッドシートにアクセスできない場合は権限設定の確認を依頼
- 見積り対象が多すぎる場合はカテゴリ別に分割して段階的に見積り
