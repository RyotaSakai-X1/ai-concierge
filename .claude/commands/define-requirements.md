# /define-requirements — 案件概要から要件定義書を作成

案件の概要情報（テキスト、URL、ファイル等）から機能要件を構造化し、要件定義書を生成します。
必要に応じて Google Sheets に要件一覧を書き出します。

## 引数

- `$ARGUMENTS` — 以下のいずれか:
  - テキスト（案件の概要説明をそのまま入力）
  - Google Docs URL（案件概要のドキュメント）
  - Google Drive フォルダ URL（関連資料が格納されたフォルダ）
  - ローカルファイルパス（案件メモ等）
  - `/extract-requirements` の出力ファイルパス（抽出済み要件をベースに定義を深掘り）

## 実行手順

### Step 1: 入力ソースの取得

引数の形式に応じて分岐する:

| 入力形式 | 取得方法 |
|---------|---------|
| テキスト（URL・パスでない文字列） | そのまま分析対象として使用 |
| Google Docs URL | `mcp__google-workspace__get_doc_as_markdown` で取得 |
| Google Drive フォルダ URL | `mcp__google-workspace__list_drive_items` → 各ファイルを取得 |
| ローカルファイルパス | Read ツールで取得 |

Google Drive アクセスでエラーが発生した場合は、ローカルファイルでの代替入力を提案する。

### Step 2: 案件情報の分析

取得した情報から以下を分析する:

1. **プロジェクトの背景・目的**: なぜこのプロジェクトが必要か
2. **ユーザーの課題**: 現状の困りごと、ペインポイント
3. **ユーザーの要望**: 「こうしたい」「欲しい」という明示的な要望
4. **暗黙の要件**: 文脈から推測される要件（未確認事項として明記）
5. **既存システム情報**: 現在使っているツール・システム
6. **制約条件**: 予算、期限、技術スタック等の制約

### Step 3: 機能要件の構造化

分析結果をもとに機能要件を構造化する:

1. 課題・要望を整理し、それぞれに ID を付与する（C1, C2... / W1, W2...）
2. 課題・要望から機能要件を導出し、ID を付与する（F1, F2...）
3. 各機能に以下を決定する:

| 項目 | 決定方法 |
|------|---------|
| 優先度（高/中/低） | ソース内での強調度合い、繰り返し言及の回数から判断。判断が難しい場合は「中」 |
| カテゴリ | 画面 / API / バッチ / 認証 / 外部連携 / その他 |
| 対応する課題/要望 | トレーサビリティを確保するため必ず紐づける |

4. 非機能要件を抽出する（パフォーマンス、セキュリティ、可用性等）
5. 優先度マトリクスに分類する（Must Have / Should Have / Nice to Have）

### Step 4: 要件定義書の生成

`knowledge/templates/requirements.md` のテンプレートに従って要件定義書を生成する。

出力先: `docs/outputs/requirements/YYYY-MM-DD-{slug}.md`
- `YYYY-MM-DD` は今日の日付
- `{slug}` はプロジェクト名から英語の短いキーワードで生成

### Step 5: Google Sheets への書き出し

AskUserQuestion でユーザーに確認する:
- 選択肢: `新規スプレッドシートを作成して書き出す` / `既存スプレッドシートに書き出す（URLを入力）` / `スプレッドシートへの書き出しはスキップ`

#### 新規スプレッドシート作成の場合

1. `mcp__google-workspace__create_spreadsheet` でスプレッドシートを作成（タイトル: `{プロジェクト名} 要件定義`）
2. 以下の3シートを作成:

**シート1: 機能要件**

ヘッダー行のカラム: `#` / `機能名` / `概要` / `対応する課題/要望` / `優先度` / `カテゴリ` / `工数（人日）` / `備考`

- 工数列は空欄のまま（`/estimating` で後から記入するため）

**シート2: 非機能要件**

ヘッダー行のカラム: `#` / `カテゴリ` / `要件` / `根拠`

**シート3: 課題・要望**

ヘッダー行のカラム: `#` / `種別`（課題/要望） / `内容` / `深刻度/優先度` / `根拠`

3. `mcp__google-workspace__modify_sheet_values` で各シートにデータを書き出す

#### 既存スプレッドシートの場合

1. ユーザーから URL を受け取り、スプレッドシート ID を抽出
2. `mcp__google-workspace__get_spreadsheet_info` でシート構造を確認
3. 適切なシート・列にデータを書き出す

### Step 6: 結果の報告

以下を報告する:

- **出力ファイル**: 要件定義書のパス
- **抽出結果**: 課題 X 件、要望 X 件、機能要件 X 件、非機能要件 X 件
- **スプレッドシート**: 書き出し先の URL（書き出した場合）
- **未確認事項**: クライアント確認が必要な項目数

推奨する次のアクション:

| 状況 | 推奨 |
|------|------|
| 画面系の要件が多い場合 | `/generate-screens` で画面設計に進む |
| 工数見積もりが必要な場合 | `/estimating` でスプレッドシートから見積もり |
| 技術設計が必要な場合 | `/generating-spec` で技術設計に進む |

## /extract-requirements との使い分け

| 観点 | /extract-requirements | /define-requirements |
|------|----------------------|---------------------|
| 入力 | ミーティング録音・議事録 | 案件概要・企画書・ヒアリングメモ |
| 目的 | 生データからの情報抽出 | 要件の構造化・定義 |
| 出力 | 種別に応じた抽出レポート | 要件定義書 + Google Sheets |
| 使い方 | まず extract で情報を抽出し、その出力を define の入力にする |

## 注意事項

- ソースに明示されていない内容を推測で機能要件に追加しない。推測が必要な場合は「未確認事項」に記載する
- 課題・要望と機能要件のトレーサビリティを必ず確保する（F1 → C1, W1 のように紐づけ）
- 優先度はソース内の強調度合いから判断する。判断が難しい場合は「中」とする
- 見積もり系（`/estimating`）とのレーン分離を意識し、工数列は空欄で出力する
