# /define-requirements — 案件概要から要件定義書を作成

案件の概要情報から機能要件を構造化し、要件定義書を生成する。必要に応じて Google Sheets に書き出す。

## 引数

- `$ARGUMENTS` — 以下のいずれか:
  - テキスト（案件の概要説明）
  - Google Docs URL / Google Drive フォルダ URL
  - ローカルファイルパス
  - `/extract-requirements` の出力ファイルパス

## Step 1: 入力ソースの取得

| 入力形式 | 取得方法 |
|---------|---------|
| テキスト（URL・パスでない文字列） | そのまま使用 |
| Google Docs URL | `mcp__google-workspace__get_doc_as_markdown` |
| Google Drive フォルダ URL | `mcp__google-workspace__list_drive_items` → 各ファイル取得 |
| ローカルファイルパス | Read ツール |

Google Drive アクセスエラー時はローカルファイルでの代替入力を提案する。

## Step 2: 案件情報の分析

分析項目:

| 項目 |
|------|
| プロジェクトの背景・目的 |
| ユーザーの課題（「困っている」「問題がある」等のペインポイント） |
| ユーザーの要望（「こうしたい」「欲しい」等の明示的な要望） |
| 暗黙の要件（文脈から推測される要件。未確認事項として明記） |
| 既存システム情報（現在使っているツール・システム） |
| 制約条件（予算、期限、技術スタック等） |

## Step 3: 機能要件の構造化

課題・要望に ID を付与（C1, C2... / W1, W2...）し、機能要件を導出（F1, F2...）する。

各機能の属性:

| 属性 | 決定方法 |
|------|---------|
| 優先度（高/中/低） | ソース内での強調度合い・繰り返し言及の回数。判断が難しい場合は「中」 |
| カテゴリ | 画面 / API / バッチ / 認証 / 外部連携 / その他 |
| 対応する課題/要望 | トレーサビリティ確保のため必ず紐づける |

非機能要件（パフォーマンス、セキュリティ、可用性等）も抽出する。

優先度マトリクス: Must Have / Should Have / Nice to Have

## Step 4: 要件定義書の生成

テンプレート: `knowledge/templates/requirements.md`

出力先: `docs/outputs/{案件slug}/requirements.md`

## Step 5: Google Sheets への書き出し

AskUserQuestion の選択肢: `新規スプレッドシートを作成して書き出す` / `既存スプレッドシートに書き出す（URLを入力）` / `スキップ`

実行順序: `create_spreadsheet` でスプレッドシート作成 → 各シート作成 → `modify_sheet_values` でデータ書き込み

シート構成（新規作成時）:

| シート | カラム |
|--------|--------|
| 機能要件 | `#` / `機能名` / `概要` / `対応する課題/要望` / `優先度` / `カテゴリ` / `工数（人日）`（空欄） / `備考` |
| 非機能要件 | `#` / `カテゴリ` / `要件` / `根拠` |
| 課題・要望 | `#` / `種別` / `内容` / `深刻度/優先度` / `根拠` |

## Step 6: 結果の報告

報告項目:
- 出力ファイルパス
- 抽出結果（課題 X 件、要望 X 件、機能要件 X 件、非機能要件 X 件）
- スプレッドシート URL（書き出した場合）
- 未確認事項の件数

次のアクション:

| 状況 | 推奨 |
|------|------|
| 画面系の要件が多い | `/generate-screens` で画面設計 |
| 工数見積もりが必要 | `/estimating` でスプレッドシートから見積もり |
| 技術設計が必要 | `/generating-spec` で技術設計 |

## /extract-requirements との使い分け

| 観点 | /extract-requirements | /define-requirements |
|------|----------------------|---------------------|
| 入力 | ミーティング録音・議事録 | 案件概要・企画書・ヒアリングメモ |
| 目的 | 生データからの情報抽出 | 要件の構造化・定義 |
| 出力 | 種別に応じた抽出レポート | 要件定義書 + Google Sheets |

## 制約

- ソースに明示されていない内容を推測で機能要件に追加しない。推測が必要な場合は「未確認事項」に記載する
- 課題・要望と機能要件のトレーサビリティを必ず確保する
- 工数列は空欄で出力する（`/estimating` とのレーン分離）
