# /generating-spec — 仕様書から実装計画・技術設計を生成

引数の仕様書ファイルから、技術構成・タスク・リスクを生成する。

## 引数

- `$ARGUMENTS` — 仕様書ファイルパス（例: `docs/specs/2025-01-01-feedback-system.md`）

## Step 1: 仕様書の読み込みと分析

入力:
- `$ARGUMENTS` の仕様書
- `knowledge/business-rules.md`
- `knowledge/` 配下の全ファイル（既存ルール・技術スタック）

分析観点:

| 観点 | 確認する内容 |
|------|------------|
| 機能要件 | 仕様書から実装すべき機能を抽出する |
| 非機能要件 | パフォーマンス、可用性、セキュリティの要件を抽出する |
| 既存スタックとの互換性 | プロジェクトの既存技術スタックと矛盾しないか確認する |
| 外部依存 | 外部 API・ライブラリ・サービスの依存関係を特定する |
| データモデル | 必要なデータ構造・スキーマを特定する |

## Step 2: 技術選定

技術選定の優先順:

| 優先度 | 基準 |
|--------|------|
| 1 | `knowledge/business-rules.md` の制約に従う |
| 2 | 既存コードのスタックを優先する |
| 3 | 上記で決定できない場合は複数案を並記しトレードオフを明示する |

「決定できない」の判定基準:

| 条件 | 判定 |
|------|------|
| business-rules.md に該当技術の指定がある | 決定可能 → その技術を採用 |
| 既存コードで同種の技術が使われている | 決定可能 → 既存と同じ技術を採用 |
| 上記いずれにも該当しない | 決定不可 → 複数案を並記し AskUserQuestion で確認 |

### 複数案がある場合の確認

AskUserQuestion で以下の形式で提示する:

| 項目 | 内容 |
|------|------|
| 質問 | 「{技術領域}はどの方式を採用しますか？」 |
| 選択肢 | 各案の名称（例: `PostgreSQL` / `MongoDB` / `SQLite`） |
| 各選択肢の description | トレードオフの要約 |

## Step 3: 出力ファイル生成

出力先: `docs/outputs/{spec-slug}/`

| ファイル名 | 内容 | 生成ルール |
|---|---|---|
| `architecture.md` | 技術構成・データフロー図 | 技術スタック一覧テーブル + Mermaid 形式のデータフロー図 + コンポーネント間の依存関係 |
| `tasks.md` | 実装タスクの詳細チェックリスト | 1タスク = 1機能 or 1ファイル単位。依存順に並べ、各タスクに影響ファイルを記載 |
| `risks.md` | リスク一覧と対策 | 影響度（高/中/低）× 発生確率（高/中/低）のマトリクスで整理 |

## エラー対応

| エラー種別 | 対応 |
|-----------|------|
| 仕様書ファイルが存在しない | エラーメッセージを表示して終了 |
| 仕様書のフォーマットが不正 | 読み取れた範囲で処理し `> ⚠️ 仮定：〜` で不明点を明示 |
| 仕様が曖昧で技術判断できない | 仮定を置いて進め `> ⚠️ 仮定：〜` で明示。重大な曖昧さは AskUserQuestion で確認（`仮定のまま進める` / `仕様の明確化を待つ`） |
| knowledge/ 配下にビジネスルールがない | 汎用的なベストプラクティスで判断し `> ⚠️ 仮定：〜` で明示 |

## 制約

- 出力は必ず `docs/outputs/{spec-slug}/` 配下に生成する
- 仕様書に記載のない機能を勝手に追加しない
- 技術選定で意見が分かれる場合はユーザーに確認する（自己判断で決定しない）
- タスクの粒度は 1 タスク = 半日〜1 日で完了できる単位を目安にする
