# /parallel-work — 複数イシューの並列実行

複数のイシュー番号を受け取り、依存関係をチェックした上で Agent Teams + worktree で並列実行する。

## 引数

- `$ARGUMENTS` — スペース区切りのイシュー番号（例: `12 14 16`）

## Step 1: 依存関係チェック

入力: 各イシューの `gh issue view {番号} --json title,body,labels,assignees`

分析項目:
- `depends on #XX` / `blocked by #XX` の有無
- 影響ファイルの推定と重複チェック

分類:

| 分類 | 条件 | 対応 |
|------|------|------|
| 独立 | 依存・競合なし | 並列実行可能 |
| 依存あり | イシュー間に依存関係 | 順次実行（依存元を先に） |
| 競合リスク | 同じファイルを変更する可能性 | 並列実行可能だがマージ時にコンフリクトの可能性 |

## Step 2: 実行計画の提示と承認

分析結果（並列実行 / 順次実行 / 競合リスク）をユーザーに提示する。

AskUserQuestion の選択肢: `実行する` / `計画を修正する` / `キャンセル`

## Step 3: チーム作成とメンバー実行

各メンバーのタスク:
- `git worktree add` で独自 worktree を作成
- ブランチ名: `feature/YYYY-MM-DD-{slug}`
- 実装開始前にファイル宣言を共有タスクリストに登録
- `/work-on-issue {番号}` 相当の実装フロー（コミットまで）を実行

テックリードの監視:
- ファイルロック宣言の重複をチェック
- 重複時は担当分担の調整 or 一方を待機させる

> メンバーの責務は「コミットまで」。プッシュと PR 作成はテックリードがレビュー後に行う。

## Step 4: テックリードによるレビュー

各メンバーの worktree で `git diff main..HEAD` を確認し、要件充足・品質をチェックする。

コンフリクト検出: 各ブランチの `git diff --name-only main..HEAD` を比較し、重複ファイルを検出。

## Step 5: ユーザーへの報告と承認

報告内容:
- 各イシューの実装サマリ（変更内容・ファイル・レビュー結果）
- コンフリクトリスク
- 推奨マージ順序

AskUserQuestion の選択肢: `全て承認して PR 作成` / `一部修正が必要` / `キャンセル`

## Step 6: PR 作成・プッシュ

各ブランチについて `git push` → `gh pr create`（`Closes #XX` を含む）→ イシューステータスを **In review** に変更。

依存ありのイシューは並列実行完了後に順次実行する。

## Step 7: クリーンアップ

- チームセッションを終了
- 各メンバーの worktree を `git worktree remove` / `git worktree prune` でクリーンアップ
- 最終レポートをユーザーに提示

## 制約

- 推奨チームサイズ: 3〜5 メンバー（トークンコスト考慮）
- メンバーでエラーや想定外の状態が発生した場合、自力リカバリせず処理を中断しテックリードに報告
- 大規模な変更を含むイシューは単独実行を推奨
