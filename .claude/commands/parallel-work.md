# /parallel-work — 複数イシューの並列実行

複数のイシュー番号を受け取り、依存関係をチェックした上で、Agent Teams のチームメンバーとして並列実行します。各メンバーは独自の worktree で作業し、共有タスクリスト経由で通信します。

## 引数

- `$ARGUMENTS` — スペース区切りのイシュー番号（例: `12 14 16`）

## 実行手順

### Step 1: イシュー情報の取得

各イシューについて `gh issue view {番号} --json title,body,labels,assignees` で情報を取得する。

### Step 2: 依存関係チェック

イシュー間の依存関係を分析する:

1. イシュー本文に `depends on #XX` や `blocked by #XX` がないか確認
2. 各イシューの影響ファイルを推定し、ファイル競合の可能性を検出
3. 結果を以下に分類:
   - **独立**: 並列実行可能
   - **依存あり**: 順次実行が必要（依存元を先に実行）
   - **競合リスク**: 並列実行可能だが、マージ時にコンフリクトの可能性あり

### Step 3: 実行計画の提示

分析結果をユーザーに提示し、AskUserQuestion で承認を得る。以下の情報を表示する:

- **並列実行（独立）**: 並列で処理するイシューの番号とタイトル
- **順次実行（依存あり）**: 依存関係があるイシューの番号・タイトル・依存先
- **競合リスク**: 同じファイルを変更する可能性のあるイシューの組み合わせ

選択肢: `実行する` / `計画を修正する` / `キャンセル`

### Step 4: チーム作成

Agent Teams でチームを作成する。各メンバーに担当イシューを割り当てる。

各メンバーのタスクには以下を含める:
- 担当イシュー番号とタイトル
- `git worktree add` で独自 worktree を作成する指示
- ブランチ命名規則: `feature/YYYY-MM-DD-{slug}`
- 実装開始前にファイル宣言を行う指示
- `/work-on-issue {番号}` 相当の実装フローを実行する指示

### Step 5: メンバー間ファイル競合回避

チーム稼働後、テックリードは以下を監視する:

1. 各メンバーのファイルロック宣言を共有タスクリストで確認する
2. 宣言が重複した場合、該当メンバーに調整指示を出す:
   - ファイルの担当分担を決める
   - または一方のメンバーを先に実行し、他方を待機させる
3. 調整完了後、各メンバーに実装続行を指示する

### Step 6: メンバー実装

各チームメンバーが worktree 上で以下を実行する:

1. イシュー分析
2. 実装
3. セルフレビュー
4. コミット（プッシュはしない）
5. 結果を共有タスクリストに報告（コミットハッシュ、変更ファイル一覧）

> メンバーの責務は「コミットまで」。プッシュと PR 作成はテックリードがレビュー後に行う。

### Step 7: テックリードによるレビュー

全メンバーの実装完了を待ち、テックリードが各メンバーの成果をレビューする:

1. 各メンバーの worktree に移動し、`git diff main..HEAD` で変更内容を確認する
2. イシューの要件を満たしているか、品質に問題がないかをチェックする
3. 問題があればメンバーに修正を指示し、修正完了を待つ
4. コンフリクト検出: 各ブランチで `git diff --name-only main..HEAD` を実行し、変更ファイルを比較する
5. 重複ファイルがあればコンフリクトリスクを記録する

### Step 8: ユーザーへの報告と承認

レビュー結果をユーザーに報告し、AskUserQuestion で承認を得る:

- **各イシューの実装サマリ**: 変更内容・変更ファイル・レビュー結果
- **コンフリクトリスク**: 共有ファイルがある場合の警告
- **推奨マージ順序**: コンフリクトリスクを考慮した順序

選択肢: `全て承認して PR 作成` / `一部修正が必要` / `キャンセル`

### Step 9: PR 作成・プッシュ

ユーザーの承認後、テックリードが各ブランチについて以下を実行する:

1. `git push -u origin {ブランチ名}` でプッシュ
2. `gh pr create --title "PRタイトル" --body "PR本文" --base main --assignee @me` で PR 作成
3. PR 本文に `Closes #XX` を含めてイシューと紐づける

### Step 10: 順次実行（依存ありのイシュー）

Step 6 の並列実行が完了し、依存元の PR がマージされた後に、依存ありのイシューを順次実行する。

### Step 11: チーム & worktree クリーンアップ

1. チームセッションを終了する
2. 各メンバーの worktree をクリーンアップする（`git worktree remove` または `git worktree prune`）
3. 最終レポートをユーザーに提示する

## 注意事項

- 推奨チームサイズ: 3〜5 メンバー（リソース消費・トークンコストを考慮）
- Agent Teams 使用時はトークン消費が増加する（各メンバーが独立したコンテキストを持つため）
- メンバーでエラーや想定外の状態が発生した場合、自力でリカバリせず処理を中断しテックリードに状況を報告する。テックリードが対応方針を判断・指示する
- 大規模な変更を含むイシューは並列実行ではなく単独実行を推奨
