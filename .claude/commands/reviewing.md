# /reviewing — 仕様書・成果物・PR のレビュー

チェックリストに基づいてレビューし、問題点や改善案を提示する。

## 引数

- `$ARGUMENTS` — レビュー対象のパスまたは PR 番号

## Step 1: レビュー種別の判定

| 引数の形式 | レビュー種別 | チェックリスト |
|-----------|------------|---------------|
| `docs/specs/...` | 仕様書レビュー | `knowledge/review/spec-checklist.md` |
| `docs/outputs/...` | 成果物レビュー | `knowledge/review/output-checklist.md` |
| `#XX` または PR 番号 | PR コードレビュー | 下記 PR レビュー手順 |

判定できない場合はユーザーに確認する。

### PR コードレビューの場合

入力: `gh pr view {番号}` + `gh pr diff {番号}`

レビュー観点:

| 観点 | 内容 |
|------|------|
| 要件充足 | PR 本文に記載された目的を達成しているか |
| コード品質 | 命名、構造、重複、可読性 |
| セキュリティ | 秘密情報、インジェクション、OWASP Top 10 |
| テスト | テストの有無と十分性 |
| 影響範囲 | 意図しない副作用がないか |

結果を `gh pr review {番号} --comment` で PR にコメントする。以降の Step は仕様書/成果物レビューに適用。

## Step 2: レビュー実行

入力:
- 対象ファイル（`$ARGUMENTS`）
- チェックリスト（Step 1 で判定）
- `knowledge/business-rules.md`
- 成果物レビュー時は対応する仕様書（`docs/specs/*-{slug}.md`）

各チェック項目の判定:

| マーク | 意味 |
|--------|------|
| ✅ | 問題なし |
| ⚠️ | 軽微な問題あり（改善推奨） |
| ❌ | 重大な問題あり（修正必須） |

## Step 3: 総合判定

| 条件 | 総合判定 |
|------|---------|
| ❌ が 0 件、⚠️ が 2 件以下 | ✅ 承認 |
| ❌ が 0 件、⚠️ が 3 件以上 | ⚠️ 条件付き承認 |
| ❌ が 1 件以上 | ❌ 差し戻し |

## Step 4: レビュー結果の出力

テンプレート: `knowledge/templates/review-report.md`

保存先:

| 対象 | 保存先 |
|------|--------|
| 仕様書 | `docs/specs/review-{元ファイル名}.md` |
| 成果物 | `docs/outputs/{slug}/review-report.md` |

## Step 5: 次のアクション提示

| 判定 | 次のアクション |
|------|---------------|
| ✅ 承認 | 仕様書 → `/executing-plan` で実行 / 成果物 → `/creating-pr` で PR 作成 |
| ⚠️ 条件付き承認 | 指摘箇所の修正案を提示し、再レビューを提案 |
| ❌ 差し戻し | 問題の原因と修正の方向性を提示 |

## 制約

- レビューは AI による自動チェック。最終判断はユーザーが行う
- ❌ 差し戻しでもユーザーが「このまま進める」と判断すれば従う
- チェックリストにない観点でも明らかな問題は指摘する
