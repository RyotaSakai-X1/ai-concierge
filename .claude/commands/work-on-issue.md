# /work-on-issue — イシュー起点の実装〜PR作成

イシュー番号を受け取り、分析 → ブランチ作成 → 実装 → セルフレビュー → PR作成を一気通貫で実行します。

スタンドアロン実行と、`/parallel-work` からチームメンバーとして呼び出される両モードに対応します。

## 引数

- `$ARGUMENTS` — イシュー番号（例: `12`）

## 実行手順

### Step 1: イシュー分析

1. `gh issue view $ARGUMENTS --json title,body,labels,assignees,milestone` でイシュー内容を取得
2. イシュー本文から以下を抽出:
   - **ゴール**: 何を達成するか
   - **やること**: 具体的なタスク一覧
   - **影響範囲**: どのファイル・機能に影響するか
3. 不明点があれば仮定を置いて進める（`> ⚠️ 仮定：〜` で明示）

### Step 2: 実行環境の判定

`git rev-parse --show-toplevel` で worktree 内で実行されているかを判定する。

- **通常実行**（メインリポジトリ内）: Step 3 でブランチ作成
- **チームメンバー実行**（`/parallel-work` からチームメンバーとして呼ばれた場合）: 既に worktree + ブランチ上なのでブランチ作成をスキップ
- **worktree 内実行**（従来の Agent tool フォールバック時）: 既にブランチ上なのでブランチ作成をスキップ

### Step 3: ブランチ作成（通常実行時のみ）

1. `.claude/rules/git-workflow.md` のブランチ命名規則に従う
2. ブランチ名: `feature/YYYY-MM-DD-{slug}`（イシューの内容から slug を決定）
3. `git checkout -b feature/YYYY-MM-DD-{slug}` を実行する

### Step 4: 実装

1. **チームメンバー実行時のみ**: 実装開始前に変更予定ファイル一覧を共有タスクリストに宣言する。他メンバーの宣言と重複がある場合はテックリードの調整指示を待つ
2. イシューの「やること」に従い、順次実装する
3. 各タスクを完了したら、チェックリストの `[ ]` を `[x]` に更新する意識で進める
4. 実装中に設計判断が必要な場合は、簡潔にコメントとして記録する

### Step 5: セルフレビュー

実装完了後、以下の観点でセルフレビューを行う:

1. `git diff` で変更内容を確認
2. チェック項目:
   - [ ] イシューの要件をすべて満たしているか
   - [ ] 不要なファイルが含まれていないか
   - [ ] コードの品質（命名、構造、重複）
   - [ ] セキュリティリスク（秘密情報、インジェクション等）
3. 問題があれば修正してから次へ進む

### Step 6: PR作成

1. `git add -A` で変更をステージングし、`git commit -m "コミットメッセージ（日本語）"` でコミットする
2. `git push -u origin {ブランチ名}` でプッシュする
3. `gh pr create --title "PRタイトル" --body "PR本文" --base main --assignee @me` でPRを作成する
   - PR本文に `Closes #XX` を含めてイシューと紐づける
   - セルフレビュー結果のサマリを PR 本文に含める

### Step 7: 結果レポート

`knowledge/templates/work-on-issue-report.md` のフォーマットに従い、実行結果を出力する。

- 通常実行時: チャットに表示
- チームメンバー実行時: 共有タスクリスト経由で結果を報告（PR URL、変更ファイル一覧を含む）
- フォールバック（従来 Agent tool）実行時: 戻り値として返却

## 注意事項

- 危険な操作（force push, ブランチ削除等）は絶対に行わない
- `.env` や秘密情報を含むファイルがあれば警告する
- 実装が大きすぎる場合は、最初のイテレーションとして最小限の変更に留め、残りを別イシューとして提案する
