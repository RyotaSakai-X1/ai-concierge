# /work-on-issue — イシュー起点の実装〜PR 作成

イシュー番号を受け取り、分析 → 実装 → セルフレビュー → PR 作成を一気通貫で実行する。スタンドアロン実行と `/parallel-work` からのチームメンバー実行に対応。

## 引数

- `$ARGUMENTS` — イシュー番号（例: `12`）

## 動作モード

| モード | 判定条件 | ブランチ作成 | プッシュ・PR 作成 |
|--------|---------|------------|-----------------|
| 通常実行 | worktree でない かつ main ブランチ | する | する |
| 既存ブランチ上 | worktree でない かつ main 以外 | スキップ（現ブランチで続行） | する |
| チームメンバー実行 | worktree 内（`/parallel-work` 経由） | スキップ | スキップ（テックリードが行う） |

判定方法: `git rev-parse --git-dir` と `git rev-parse --git-common-dir` が異なれば worktree 内。

## Step 1: イシュー分析

入力: `gh issue view $ARGUMENTS --json title,body,labels,assignees,milestone`

エラー時対応:

| エラー | 対応 |
|--------|------|
| イシュー番号が未指定 | 処理中断。「イシュー番号を指定してください（例: `/work-on-issue 12`）」と表示 |
| イシュー取得失敗（API エラー・番号不正） | 処理中断。エラー内容を表示 |

イシュー取得成功時の即時処理:

| アクション | コマンド |
|-----------|---------|
| イシューアサイン | `gh issue edit $ARGUMENTS --add-assignee @me` |
| ステータス変更 | `.claude/rules/git-workflow.md` の手順で **In progress** に変更 |

抽出項目:

| 項目 | 抽出アクション |
|------|--------------|
| ゴール | イシュー本文から達成条件を特定する |
| やること | 「やること」セクションのチェックリストを抽出する |
| 影響範囲 | チェックリストから変更対象ファイル・機能を列挙する |
| 依存関係 | 「関連イシュー」セクションから前提イシューを特定する |

出力: 抽出した 4 項目のサマリ

不明点は仮定を置いて進める（`> ⚠️ 仮定：〜` で明示）。

## Step 2: ブランチ作成（通常実行時のみ）

ブランチ名: `feature/YYYY-MM-DD-{slug}`（`.claude/rules/git-workflow.md` の命名規則に従う）

| アクション | コマンド |
|-----------|---------|
| ブランチ作成 | `git checkout -b feature/YYYY-MM-DD-{slug}` |

## Step 3: 実装

入力: Step 1 の抽出結果（やることリスト、影響範囲）

チームメンバー実行時の事前処理:

| アクション | 内容 |
|-----------|------|
| ファイルロック宣言 | 変更予定ファイル一覧を共有タスクリストに登録する |
| 競合確認 | 他メンバーの宣言と重複がないか確認する |

実装ルール:

| ルール | 内容 |
|--------|------|
| 実装順序 | 「やること」チェックリストの上から順に実装する |
| 完了基準 | チェックリストの全項目が完了していること |
| スコープ | イシューに記載された範囲のみ変更する。範囲外の改善は別イシューとして提案する |

出力: 変更されたファイル一覧

中断条件:

| 条件 | 対応 |
|------|------|
| 実行時エラー（コマンド失敗、ファイル不整合） | 処理中断。エラー内容をテックリードに報告 |
| 想定外の状態（ブランチ不一致、未追跡の変更検出） | 処理中断。状態をテックリードに報告 |
| 判断に必要な情報の不足（仕様未定義、曖昧な要件） | 処理中断。不足情報をテックリードに報告 |

## Step 4: セルフレビュー

`/reviewing` を使い、変更内容をレビューする。

| レビュー対象 | 入力 |
|------------|------|
| 変更差分 | `git diff`（未コミットの場合） |

`/reviewing` の判定結果に応じた対応:

| 判定 | 対応 |
|------|------|
| ✅ 承認 | Step 5 へ進む |
| ⚠️ 条件付き承認 | 指摘箇所を修正し、再度 `/reviewing` |
| ❌ 差し戻し | 指摘箇所を修正し、再度 `/reviewing` |

## Step 5: コミット・PR 作成

入力: Step 4 のレビュー結果、変更差分

通常実行・既存ブランチ時のユーザー確認（AskUserQuestion）:

| 表示内容 | 選択肢 |
|---------|--------|
| 変更概要 + コミットメッセージ案を提示 | `コミットして PR 作成する` / `コミットメッセージを修正する` / `実装を修正する` |

| モード | 処理 |
|--------|------|
| チームメンバー実行 | `git add -A` → `git commit` → コミットハッシュと変更ファイル一覧を共有タスクリストに報告して終了 |
| 通常実行・既存ブランチ | `/creating-pr` に委譲（PR 本文に `Closes #$ARGUMENTS` + セルフレビューサマリを含める） |

## Step 6: 結果レポート

テンプレート: `knowledge/templates/work-on-issue-report.md`

| モード | 報告先 |
|--------|--------|
| 通常実行 | チャットに表示 |
| チームメンバー実行 | 共有タスクリスト経由（コミットハッシュ、変更ファイル一覧を含む） |

## Step 7: PR 作成後の確認（通常実行・既存ブランチ時のみ）

AskUserQuestion で修正の有無を確認する:

| 表示内容 | 選択肢 |
|---------|--------|
| PR の URL を提示 | `修正がある` / `修正はない` |

| 選択 | 対応 |
|------|------|
| `修正がある` | このブランチで修正作業を継続 → 修正後 Step 4（セルフレビュー）に戻る |
| `修正はない` | Step 8 へ |

## Step 8: クリーンアップ確認（通常実行・既存ブランチ時のみ）

AskUserQuestion でクリーンアップを確認する:

| 表示内容 | 選択肢 |
|---------|--------|
| 作業完了 | `ブランチを削除して main に戻る` / `このブランチで続ける` |

| 選択 | 対応 |
|------|------|
| `ブランチを削除して main に戻る` | ローカルブランチ削除 → `git checkout main && git pull` → 次のイシュー提案 |
| `このブランチで続ける` | 終了 |

### main に戻った後

`gh issue list` で優先度が高いものから確認し、次のイシューを提案する:

| 表示内容 | 選択肢 |
|---------|--------|
| 次にやるべきイシュー | `{イシュータイトル}` / `他のイシューを見る` / `今は作業しない` |

## 制約

- force push・ブランチ削除は行わない
- `.env` や秘密情報を含むファイルがあれば警告して除外する
- 実装が大きすぎる場合は最小限の変更に留め、残りを別イシューとして提案する
- コミットメッセージは日本語で記述する
