# /generate-screens — 画面一覧・画面遷移図・レイアウトの自動生成

要件定義の出力やテキスト指示から、画面一覧・画面遷移図（Mermaid）・各画面の概要レイアウトを自動生成します。

## 引数

- `$ARGUMENTS` — 以下のいずれか:
  - `/extract-requirements` の出力ファイルパス（例: `docs/outputs/requirements/2026-03-11-xxx.md`）
  - テキストでの直接指示（例: `ダッシュボード画面とユーザー管理画面を設計して`）

## 実行手順

### Step 1: 入力の取得と分析

#### パターン A: ファイルパスが指定された場合

1. Read ツールで指定ファイルを読み込む
2. ファイルから以下を抽出:
   - 機能要件一覧（画面に紐づく機能）
   - ユーザーの操作フロー
   - 表示すべきデータ
   - ユーザーロール（権限による画面分岐があれば）

#### パターン B: テキスト指示の場合

1. 指示内容から必要な画面と機能を分析
2. 不明な点は合理的な仮定を置き、`> ⚠️ 仮定：` で明示する

### Step 2: 画面設計

`knowledge/templates/prototype.md` のテンプレートに従い、以下を設計する。

#### 2-1: 画面一覧の作成

要件から必要な画面を洗い出し、以下の観点で整理する:

- **画面の粒度**: 1つの主要機能 = 1画面を基本とする。CRUD がある場合は一覧/詳細/作成・編集の3画面に分離する
- **共通画面**: ログイン、ダッシュボード、設定、エラー画面など、要件に明示されていなくても一般的に必要な画面は含める
- **主要操作**: 各画面で行える操作（閲覧/作成/編集/削除/検索/フィルタ等）
- **表示データ**: 各画面で表示するデータ項目

各画面を以下のカラムで表にする: `#` / `画面名` / `目的` / `主要操作` / `表示データ` / `備考`

#### 2-2: 画面遷移図の作成

Mermaid の `graph TD` 形式で画面遷移図を作成する:

- ノードは画面名（日本語可）
- エッジのラベルにはユーザーアクション（クリック、送信等）を記載
- ログイン → メイン画面の導線を起点にする
- 共通ナビゲーション（ヘッダーメニュー等）からの遷移は注釈として記載する

#### 2-3: 各画面のレイアウト概要

各画面について以下を記述する:

- **画面の目的**: 1文で記述
- **レイアウト構成**: ヘッダー / サイドバー / メインコンテンツ / フッターの各エリアの内容を表にする
- **主要コンポーネント**: 画面を構成する UI コンポーネント（検索バー、データテーブル、フォーム、モーダル等）を箇条書き
- **ユーザーアクション**: アクション / トリガー / 結果を表にする

### Step 3: 出力ファイルの保存

以下のディレクトリ構成で保存する:

| パス                                          | 内容                     |
| --------------------------------------------- | ------------------------ |
| `docs/outputs/prototypes/YYYY-MM-DD-{slug}/`  | 出力ルートディレクトリ   |
| `screens.md`                                  | 画面一覧                 |
| `flow.md`                                     | 画面遷移図（Mermaid）    |
| `layouts/01-{画面名slug}.md`                  | 各画面のレイアウト概要   |
| `layouts/02-{画面名slug}.md`                  | （画面数分繰り返し）     |

- `YYYY-MM-DD` は今日の日付
- `{slug}` はプロジェクト名から生成（英数字・ハイフン）
- 各ファイルの先頭にプロジェクト名・作成日・バージョンを記載する

#### screens.md の構成

テンプレートの「基本情報」と「画面一覧」セクションを含む。

#### flow.md の構成

テンプレートの「基本情報」と「画面遷移図」セクションを含む。Mermaid コードブロックで遷移図を記述する。

#### layouts/{nn}-{slug}.md の構成

テンプレートの「各画面レイアウト」セクションの形式に従い、1画面1ファイルで記述する。ファイル名の番号は画面一覧の `#` に対応させる。

### Step 4: 結果の報告

以下の形式でユーザーに報告する:

- **出力先**: ディレクトリパス
- **画面数**: 生成した画面の数
- **ファイル一覧**: 生成した全ファイルのリスト

次のアクションを提案する:
- `/wireframe` で FigJam にワイヤーフレームを生成
- `/generating-spec` で技術設計に進む
- レイアウトの修正・追加が必要であれば指示を受ける

## 設計指針

- **ユーザー中心**: 要件のユーザーアクションを起点に画面を設計する
- **一貫性**: 画面間のナビゲーション・レイアウトパターンを統一する
- **最小限**: 要件に明示されていない高度な機能（リアルタイム通知、ドラッグ&ドロップ等）は含めない。必要な場合は備考に記載する
- **標準パターンの活用**: 一覧 → 詳細 → 編集、ダッシュボード → 各機能画面などの標準的な画面遷移パターンを活用する

## 注意事項

- 要件にない画面を勝手に追加しない。一般的に必要な画面（ログイン、エラー等）のみ例外として追加し、その旨を備考に記載する
- Mermaid の画面遷移図は読みやすさを優先し、画面数が多い場合は機能ブロックごとにサブグラフで分割する
- レイアウトはワイヤーフレームレベルの概要に留める。ピクセル単位の詳細やカラー指定は `/design` コマンドの領域
- 仮定を置いた場合は必ず `> ⚠️ 仮定：` で明示する

## 関連コマンド

- `/extract-requirements` — この入力となる要件定義書を生成する
- `/wireframe` — この出力を元に FigJam ワイヤーフレームを生成する
- `/generating-spec` — この出力を元に技術設計を行う
- `/design` — デザインシステム準拠の UI コードを生成する
