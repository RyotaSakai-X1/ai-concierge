# /generate-screens — 画面一覧・画面遷移図・レイアウトの自動生成

要件定義の出力やテキスト指示から、画面一覧・画面遷移図（Mermaid）・各画面のレイアウト概要を生成する。

## 引数

- `$ARGUMENTS` — 以下のいずれか:
  - `/extract-requirements` または `/define-requirements` の出力ファイルパス
  - テキストでの直接指示

## Step 1: 入力の取得と分析

| 入力形式 | 取得方法 |
|---------|---------|
| ファイルパス | Read ツールで取得 |
| テキスト指示 | そのまま分析。不明な点は `> ⚠️ 仮定：` で明示 |

ファイルからの抽出対象:
- 機能要件一覧（画面に紐づく機能）
- ユーザーの操作フロー
- 表示すべきデータ
- ユーザーロール（権限による画面分岐があれば）

## Step 2: 画面設計

テンプレート: `knowledge/templates/prototype.md`

### 画面一覧

画面の粒度: 1 つの主要機能 = 1 画面。CRUD がある場合は一覧 / 詳細 / 作成・編集の 3 画面に分離。

共通画面（ログイン、ダッシュボード、設定、エラー画面等）は要件に明示されていなくても含める。

各画面のカラム: `#` / `画面名` / `目的` / `主要操作` / `表示データ` / `備考`

### 画面遷移図

Mermaid `graph TD` 形式。ノードは画面名（日本語）、エッジのラベルにユーザーアクションを記載。ログイン → メイン画面を起点とする。画面数が 10 超の場合は subgraph で分割。

### 各画面のレイアウト概要

各画面に含める項目:

| 項目 | 内容 |
|------|------|
| 画面の目的 | 1 文で記述 |
| レイアウト構成 | ヘッダー / サイドバー / メインコンテンツ / フッターの各エリア内容をテーブルで |
| 主要コンポーネント | 検索バー、データテーブル、フォーム、モーダル等を箇条書き |
| ユーザーアクション | アクション / トリガー / 結果 をテーブルで |

## Step 3: 出力ファイルの保存

出力先: `docs/outputs/{案件slug}/screens/`

| ファイル | 内容 |
|---------|------|
| `screens.md` | 画面一覧（テンプレートの「基本情報」+「画面一覧」） |
| `flow.md` | 画面遷移図（テンプレートの「基本情報」+「画面遷移図」、Mermaid コードブロック） |
| `layouts/{nn}-{画面名slug}.md` | 各画面のレイアウト概要（1 画面 1 ファイル、番号は画面一覧の `#` に対応） |

各ファイルの先頭にプロジェクト名・作成日・バージョンを記載する。

## Step 4: 結果の報告

報告項目: 出力先ディレクトリパス / 画面数 / ファイル一覧

次のアクション:

| 推奨 |
|------|
| `/wireframe` で FigJam にワイヤーフレーム生成 |
| `/generating-spec` で技術設計 |
| レイアウトの修正・追加が必要であれば指示を受ける |

## 制約

- 要件にない画面を追加しない。一般的に必要な画面（ログイン、エラー等）のみ例外（備考に記載）
- Mermaid の遷移図は読みやすさ優先。画面数が多い場合は subgraph で分割
- レイアウトはワイヤーフレームレベルの概要に留める（ピクセル詳細やカラー指定は `/design` の領域）
- 仮定を置いた場合は `> ⚠️ 仮定：` で明示

## 関連コマンド

- `/extract-requirements` — この入力となる要件定義書を生成
- `/wireframe` — この出力を元に FigJam ワイヤーフレームを生成
- `/generating-spec` — この出力を元に技術設計を行う
