# /create-issues-from-meeting — 議事録・レポートからイシュー一括起票

議事録やレポートファイルのネクストアクション・アクションアイテムを読み取り、GitHub Issues に一括起票します。

`/extract-requirements` の出力（MTGインサイトレポート、検討レポート等）や、手動で作成した議事録にも対応します。

## 引数

- `$ARGUMENTS` — 議事録またはレポートファイルのパス（省略時は `docs/outputs/` 内の最新ファイルを自動選択）

## 実行手順

### Step 1: 対象ファイルの取得

#### 引数ありの場合

1. 指定されたファイルパスを Read ツールで読み込む

#### 引数なしの場合

1. 以下のディレクトリから最新の `.md` ファイルを自動選択する:
   - `docs/outputs/{案件slug}/meetings/`
   - `docs/outputs/{案件slug}/requirements.md`
   - `docs/outputs/{案件slug}/discussions/`
2. 複数ディレクトリに候補がある場合は AskUserQuestion でユーザーに選択を求める

### Step 2: アクションアイテムの抽出

ファイル内容を分析し、以下のセクションからアクションアイテムを抽出する:

**対象セクション（優先順）**:
- `## ネクストアクション` — テーブル形式のアクション一覧
- `## 決定事項` — チェックリスト形式の決定事項
- `## 未確認事項` / `## 残課題（継続検討）` — チェックリスト形式の課題

各アクションアイテムから以下の情報を抽出する:

| 抽出項目   | 抽出元                                             |
| ---------- | -------------------------------------------------- |
| タイトル   | アクション名 / 決定内容                             |
| 説明       | 関連する議題・背景の文脈を含めて記述                 |
| 担当者     | テーブルの「担当」列 / チェックリスト内の担当者名    |
| 期限       | テーブルの「期限」列 / 本文中の日付                  |
| 優先度     | テーブルの「優先度」列                               |
| ソース     | 元ファイルのパスとセクション名                       |

### Step 3: イシュー内容のプレビュー

起票する前に、抽出結果をユーザーにプレビュー表示する。

**プレビュー表示の形式**: `## 起票予定のイシュー（X 件）` の見出しの下に以下のテーブルを表示する:

| #  | タイトル | 担当  | 期限       | ラベル         |
| -- | -------- | ----- | ---------- | -------------- |
| 1  | ...      | @xxx  | YYYY-MM-DD | meeting-action |
| 2  | ...      | @yyy  | YYYY-MM-DD | meeting-action |

AskUserQuestion で確認する:
- 選択肢: `全件起票する` / `選択して起票する` / `キャンセル`

`選択して起票する` が選ばれた場合は、各イシューについて `起票する` / `スキップする` を選択可能にする。

### Step 4: ラベルの準備

`meeting-action` ラベルが存在するか確認する。`gh label list --search "meeting-action"` を実行し、存在しない場合は `gh label create "meeting-action" --description "ミーティングから生まれたアクションアイテム" --color "c5def5"` で作成する。

### Step 5: イシューの一括起票

承認されたアクションアイテムを `gh issue create` で起票する。

各イシューの本文は以下のセクション構成で記述する:

| セクション  | 内容                                                                 | 条件           |
| ----------- | -------------------------------------------------------------------- | -------------- |
| `## 概要`   | アクション内容の詳細                                                 | 必須           |
| `## 背景`   | 元のミーティング/レポートの文脈                                       | 必須           |
| `## ソース` | ファイルパス、抽出元セクション名、ミーティング日を箇条書きで記載       | 必須           |
| `## 期限`   | `⏰ {YYYY-MM-DD}` の形式で記載                                       | 期限がある場合 |

起票オプション:
- `--label meeting-action`
- `--assignee {担当者の GitHub ユーザー名}`（担当者が特定できる場合）

#### 担当者のマッピング

ファイル内の担当者名から GitHub ユーザー名を推定する:

1. 名前がそのまま GitHub ユーザー名の場合 → そのまま使用
2. 名前から推定できない場合 → `--assignee` を省略し、本文に「担当: {名前}」を記載

### Step 6: 結果の報告

起票結果を以下の形式で報告する:

- **起票件数**: X 件
- **イシュー一覧**: 各イシューの番号・タイトル・担当者・URL をテーブル表示
- **スキップ件数**: Y 件（ある場合）
- **ソースファイル**: 元ファイルのパス

## 注意事項

- 起票前に必ずユーザーの承認を得る（Step 3 のプレビュー確認）
- 重複イシューの可能性がある場合は警告する（既存イシューのタイトルと類似度をチェック）
- 期限は `⏰ 期限:` の形式でイシュー本文に記載する（GitHub Issues には期限フィールドがないため）
- 1回の実行で大量（10件以上）のイシューが起票される場合は、バッチごとに確認を挟む

## 関連コマンド

- `/extract-requirements` — MTGインサイトレポートや検討レポートを生成する（このコマンドの入力元）
- `/work-on-issue` — 起票されたイシューの実装に着手する
