# /create-issues-from-meeting — 議事録・レポートからイシュー一括起票

議事録やレポートファイルのアクションアイテム、および要件定義書の機能要件・非機能要件を読み取り、GitHub Issues に一括起票する。`/extract-requirements` や `/define-requirements` の出力、手動作成の議事録に対応。

## 引数

- `$ARGUMENTS` — 議事録またはレポートファイルのパス
  - 省略時は `docs/outputs/` 内の最新ファイルを自動選択

自動選択の探索先:
- `docs/outputs/{案件slug}/meetings/`
- `docs/outputs/{案件slug}/requirements.md` — 要件定義書（`/define-requirements` の出力）
- `docs/outputs/{案件slug}/discussions/`

複数候補がある場合は AskUserQuestion でユーザーに選択を求める。

## Step 1: アクションアイテム・要件の抽出

入力: `$ARGUMENTS` のファイル

対象セクション（優先順）:
- `## ネクストアクション` — テーブル形式
- `## 決定事項` — チェックリスト形式
- `## 未確認事項` / `## 残課題（継続検討）` — チェックリスト形式
- `## 機能要件一覧` — 各機能要件を実装イシューとして起票（要件定義書向け）
- `## 非機能要件` — 非機能要件をイシューとして起票（要件定義書向け）
- `## 優先度マトリクス` — 優先度情報を上記イシューのラベルに反映（要件定義書向け）

抽出項目（議事録の場合）:

| 項目 | 抽出元 |
|------|--------|
| タイトル | アクション名 / 決定内容 |
| 説明 | 関連する議題・背景の文脈 |
| 担当者 | テーブルの「担当」列 / チェックリスト内の担当者名 |
| 期限 | テーブルの「期限」列 / 本文中の日付 |
| 優先度 | テーブルの「優先度」列 |
| ソース | 元ファイルのパスとセクション名 |

抽出項目（要件定義書の場合）:

| 項目 | 抽出元 |
|------|--------|
| タイトル | 機能要件名 / 非機能要件名 |
| 説明 | 要件の詳細説明・受け入れ条件 |
| 優先度 | 優先度マトリクスの値 / 要件の優先度列 |
| ソース | 元ファイルのパスとセクション名 |

## Step 2: プレビューと承認

抽出結果をテーブルでユーザーに表示する。

表示カラム: `#` / `タイトル` / `担当` / `期限` / `ラベル`

AskUserQuestion の選択肢: `全件起票する` / `選択して起票する` / `キャンセル`

`選択して起票する` の場合は各イシューについて `起票する` / `スキップする` を選択可能にする。

## Step 3: イシュー起票

入力ファイルの種別を判定し、適切なラベルを使用する:
- **要件定義書**（`## 機能要件一覧` セクションを含む）: `enhancement` ラベルを使用
- **議事録**（`## ネクストアクション` セクションを含む）: `meeting-action` ラベルを使用

使用するラベルが存在しない場合は `gh label create` で作成する。

各イシューの本文構成:

| セクション | 内容 | 条件 |
|-----------|------|------|
| `## 概要` | アクション内容の詳細 / 要件の詳細説明 | 必須 |
| `## 背景` | 元のミーティング/レポート/要件定義書の文脈 | 必須 |
| `## 受け入れ条件` | 要件定義書に記載の受け入れ条件 | 要件定義書の場合 |
| `## ソース` | ファイルパス、抽出元セクション名、ミーティング日 | 必須 |
| `## 期限` | `⏰ {YYYY-MM-DD}` | 期限がある場合 |

起票オプション: `--label {enhancement | meeting-action}` + `--assignee {GitHub ユーザー名}`（特定できる場合）

担当者マッピング: GitHub ユーザー名と一致しない場合は `--assignee` を省略し本文に「担当: {名前}」を記載。

## Step 4: 結果の報告

報告項目:
- 起票件数
- イシュー一覧テーブル（番号 / タイトル / 担当者 / URL）
- スキップ件数（ある場合）
- ソースファイルパス

## 制約

- 起票前に必ずユーザーの承認を得る（Step 2）
- 重複イシューの可能性がある場合は警告する
- 期限は `⏰ 期限:` の形式でイシュー本文に記載する
- 10 件以上の場合はバッチごとに確認を挟む

## 関連コマンド

- `/extract-requirements` — この入力元となるレポートを生成
- `/define-requirements` — この入力元となる要件定義書を生成
- `/work-on-issue` — 起票されたイシューの実装に着手
