# /wireframe — 各画面のワイヤーフレーム（レイアウト記述）を生成

`/generate-screens` で定義した画面一覧をもとに、各画面の低忠実度レイアウト（ワイヤーフレーム）を Markdown で生成する。

## 引数

- `$ARGUMENTS` — 以下のいずれか:
  - `/generate-screens` の出力ディレクトリパス（`docs/outputs/{slug}/screens/`）
  - `/define-requirements` の出力ファイルパス（画面一覧を自動生成してからレイアウト作成）
  - テキストでの直接指示

## Step 1: 入力の取得と分析

| 入力形式 | 取得方法 |
|---------|---------|
| `/generate-screens` 出力 | Read で `screens.md` を取得し、画面一覧を抽出 |
| 要件定義書 | Read で取得し、画面に紐づく機能要件を抽出 |
| テキスト指示 | そのまま分析。不明な点は `> ⚠️ 仮定：` で明示 |

画面一覧がない場合（要件定義書やテキスト指示の場合）は、まず画面一覧を簡易生成してからレイアウトに進む。

## Step 2: 各画面のレイアウト設計

各画面に含める項目:

| 項目 | 内容 |
|------|------|
| 画面の目的 | 1 文で記述 |
| レイアウト構成 | ヘッダー / サイドバー / メインコンテンツ / フッターの各エリア内容をテーブルで |
| 主要コンポーネント | 検索バー、データテーブル、フォーム、モーダル等を箇条書き |
| ユーザーアクション | アクション / トリガー / 結果 をテーブルで |
| 遷移先 | この画面から遷移可能な画面を箇条書き |

### レイアウト記述のルール

- ピクセル指定やカラー指定はしない（ワイヤーフレームレベルに留める）
- コンポーネントは機能を記述し、見た目の詳細は `/design` の領域
- 共通レイアウト（ヘッダー、サイドバー、フッター）は `00-common.md` にまとめ、各画面では差分のみ記載
- レスポンシブ対応が必要な場合は「モバイル時の変更点」を備考に記載

## Step 3: 出力ファイルの保存

出力先: `docs/outputs/{案件slug}/wireframes/`

| ファイル | 内容 |
|---------|------|
| `00-common.md` | 共通レイアウト（ヘッダー、サイドバー、フッター） |
| `{nn}-{画面名slug}.md` | 各画面のレイアウト概要（1 画面 1 ファイル、番号は画面一覧の `#` に対応） |

各ファイルの先頭にプロジェクト名・作成日・バージョンを記載する。

## Step 4: 結果の報告

報告項目: 出力先ディレクトリパス / 画面数 / ファイル一覧

次のアクション:

| 状況 | 推奨 |
|------|------|
| ダイアグラムがまだ | `/generate-diagrams` で FigJam にフロー図を生成 |
| 工数見積もりが必要 | `/estimating` でスプレッドシートから見積もり |
| 技術設計に進む | `/generating-spec` で技術設計 |
| レビューが必要 | `/reviewing` で成果物レビュー |

## 制約

- 要件にない画面のレイアウトを作成しない
- レイアウトはワイヤーフレームレベルの概要に留める（ピクセル詳細やカラー指定は `/design` の領域）
- 仮定を置いた場合は `> ⚠️ 仮定：` で明示
- 画面一覧の `#` と番号を一致させ、トレーサビリティを確保する

## 関連コマンド

- `/generate-screens` — この入力となる画面一覧・遷移図を生成
- `/generate-diagrams` — FigJam にダイアグラムを生成
- `/generating-spec` — この出力を元に技術設計を行う
- `/reviewing` — この出力をレビュー
