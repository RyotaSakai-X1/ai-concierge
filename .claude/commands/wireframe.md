# /wireframe — 要件定義から FigJam にワイヤーフレーム図を生成

要件定義書や画面設計の出力をもとに、画面遷移図・フロー図を FigJam に自動生成します。

## 引数

- `$ARGUMENTS` — 以下のいずれか:
  - `/generate-screens` の出力ディレクトリパス（例: `docs/outputs/prototypes/2026-03-14-xxx/`）
  - `/define-requirements` または `/extract-requirements` の出力ファイルパス
  - Google Sheets URL（要件一覧が記載されたスプレッドシート）
  - テキストでの直接指示（例: `ユーザー登録〜ダッシュボードまでの画面遷移図`）

## 実行手順

### Step 1: 入力の取得と分析

引数の形式に応じて分岐する:

| 入力形式 | 取得方法 | 抽出対象 |
|---------|---------|---------|
| `/generate-screens` 出力ディレクトリ | Read ツールで `screens.md` と `flow.md` を取得 | 画面一覧と既存の Mermaid 遷移図 |
| 要件定義書（Markdown ファイル） | Read ツールで取得 | 機能要件一覧、画面に紐づく機能 |
| Google Sheets URL | `mcp__google-workspace__read_sheet_values` で取得 | 機能要件の行データ |
| テキスト指示 | そのまま分析 | ユーザーの意図する画面とフロー |

取得したデータから以下を抽出する:

1. **画面の一覧**: 必要な画面名と用途
2. **画面間の遷移**: ユーザーがどの画面からどの画面に移動するか
3. **ユーザーアクション**: 遷移のトリガーとなる操作（ボタンクリック、送信等）
4. **条件分岐**: ログイン成功/失敗、権限による分岐等

### Step 2: 生成する図の種類を決定

抽出した情報をもとに、生成する図の種類を AskUserQuestion で確認する:

| 図の種類 | 適したケース | Mermaid タイプ |
|---------|------------|---------------|
| 画面遷移図 | 画面間のナビゲーションフロー | `graph LR` |
| ユーザーフロー | 特定のタスクの手順（登録、購入等） | `graph TD` |
| 状態遷移図 | ステータスの変化（注文状態、承認フロー等） | `stateDiagram-v2` |
| シーケンス図 | ユーザー・システム間のやり取り | `sequenceDiagram` |

選択肢: `画面遷移図` / `ユーザーフロー` / `状態遷移図` / `シーケンス図` / `全て生成`

### Step 3: Mermaid コードの生成

選択された図の種類に応じて Mermaid コードを生成する。

#### 画面遷移図の場合

以下のルールで `graph LR` を構成する:

- ノードは画面名を日本語で記述し、引用符で囲む（例: `A["ログイン画面"]`）
- エッジのラベルにはユーザーアクションを記載（例: `-->|"ログインボタン"|`）
- 共通ナビゲーション（グローバルメニュー）は subgraph で表現
- 画面数が10を超える場合は機能ブロックごとに subgraph で分割
- 色は控えめに使用（エントリーポイントを緑、エラー画面を赤程度）

#### ユーザーフローの場合

`graph TD` で上から下に流れるフローを構成する:

- 開始ノードと終了ノードを丸括弧で表現
- 条件分岐はひし形 `{" "}`で表現
- 各ステップにユーザーアクションまたはシステム応答を記載

#### 状態遷移図の場合

`stateDiagram-v2` で構成する:

- 各状態をノードとして定義
- 遷移条件をラベルに記載
- 開始状態 `[*]` と終了状態 `[*]` を含める

#### シーケンス図の場合

`sequenceDiagram` で構成する:

- 参加者（ユーザー、フロントエンド、バックエンド、外部API等）を定義
- メッセージのやり取りを時系列で記述

### Step 4: FigJam に生成

`mcp__figma__generate_diagram` を使用して FigJam に図を生成する。

パラメータ:

| パラメータ | 値 |
|-----------|-----|
| name | `{プロジェクト名} {図の種類}` |
| mermaidSyntax | Step 3 で生成した Mermaid コード |
| userIntent | 図の目的を簡潔に記述 |

**重要**: ツールの戻り値に含まれる URL を必ずマークダウンリンクとしてユーザーに表示する。

「全て生成」が選択された場合は、図の種類ごとに `mcp__figma__generate_diagram` を順に呼び出す。

### Step 5: 結果の報告

以下を報告する:

- **生成した図**: 図の種類と FigJam の URL（マークダウンリンク）
- **含まれる画面数**: 図に含まれる画面/ノードの数
- **Mermaid コード**: 生成したコードをローカルにも保存する場合のパス

Mermaid コードのローカル保存先: `docs/outputs/wireframes/YYYY-MM-DD-{slug}/` 配下に図の種類ごとのファイル

| ファイル | 内容 |
|---------|------|
| `screen-flow.md` | 画面遷移図の Mermaid コード |
| `user-flow.md` | ユーザーフローの Mermaid コード |
| `state-diagram.md` | 状態遷移図の Mermaid コード |
| `sequence.md` | シーケンス図の Mermaid コード |

推奨する次のアクション:

| 状況 | 推奨 |
|------|------|
| 画面設計がまだの場合 | `/generate-screens` で画面レイアウトを設計 |
| 工数見積もりが必要な場合 | `/estimating` でスプレッドシートから見積もり |
| 技術設計に進む場合 | `/generating-spec` で技術設計を生成 |

## Mermaid コードの制約

`mcp__figma__generate_diagram` の制約に従う:

- 対応する図のタイプ: `graph`, `flowchart`, `sequenceDiagram`, `stateDiagram`, `stateDiagram-v2`, `gantt`
- graph/flowchart では全てのテキストを引用符で囲む
- 絵文字を使用しない
- `\n` で改行しない（実際の改行を使う）
- 色のスタイリングは控えめに（graph/flowchart のみ対応、`mcp__figma__generate_diagram` の制約）

## 注意事項

- 要件にない画面や遷移を勝手に追加しない。一般的に必要なもの（ログイン→メイン導線等）のみ例外
- 画面数が多い場合は subgraph で機能ブロックに分割し、読みやすさを優先する
- FigJam への生成後、レイアウトの微調整は Figma 上で行うようユーザーに案内する
- 仮定を置いた場合は `> ⚠️ 仮定：` で明示する

## 関連コマンド

- `/define-requirements` — 要件定義書を生成する（この入力になる）
- `/generate-screens` — 画面レイアウトを Markdown で設計する（この入力にもなる）
- `/generating-spec` — この出力を元に技術設計を行う
