# /wireframe — 各画面のワイヤーフレーム（レイアウト）を生成

画面一覧・要件定義をもとに、各画面の低忠実度レイアウト記述（ワイヤーフレーム）を Markdown で生成する。

## 引数

- `$ARGUMENTS` — 以下のいずれか:
  - `/generate-screens` の出力ディレクトリパス（`screens/` フォルダ）
  - `/define-requirements` または `/extract-requirements` の出力ファイルパス
  - テキストでの直接指示

## Step 1: 入力の取得と分析

| 入力形式 | 取得方法 |
|---------|---------|
| `/generate-screens` 出力 | Read で `screens.md`（画面一覧）と `flow.md`（遷移図）を取得 |
| 要件定義書（Markdown） | Read で取得し、機能要件から画面を推定 |
| テキスト指示 | そのまま分析。不明な点は `> ⚠️ 仮定：` で明示 |

抽出対象:
- 画面一覧（画面名・目的・主要操作・表示データ）
- 画面間の遷移関係
- 各画面で必要なコンポーネント

## Step 2: 各画面のワイヤーフレーム生成

画面ごとに以下の項目を含むレイアウト記述を生成する。

### 各画面に含める項目

| 項目 | 内容 |
|------|------|
| 画面の目的 | 1 文で記述 |
| レイアウト構成 | ヘッダー / サイドバー / メインコンテンツ / フッターの各エリア内容をテーブルで |
| 主要コンポーネント | 検索バー、データテーブル、フォーム、モーダル等を箇条書き |
| ユーザーアクション | アクション / トリガー / 結果 をテーブルで |
| 遷移先 | この画面から遷移可能な画面の一覧 |

### 画面の粒度

- 1 つの主要機能 = 1 画面
- CRUD がある場合は一覧 / 詳細 / 作成・編集の 3 画面に分離
- 共通画面（ログイン、ダッシュボード、設定、エラー画面等）は要件に明示されていなくても含める

## Step 3: 出力ファイルの保存

出力先: `docs/outputs/{案件slug}/wireframes/`

| ファイル | 内容 |
|---------|------|
| `layouts/{nn}-{画面名slug}.md` | 各画面のワイヤーフレーム（1 画面 1 ファイル、番号は画面一覧の `#` に対応） |

各ファイルの先頭にプロジェクト名・作成日・バージョンを記載する。

## Step 4: 結果の報告

報告項目: 出力先ディレクトリパス / 画面数 / ファイル一覧

次のアクション:

| 推奨 |
|------|
| `/generate-diagrams` で FigJam にダイアグラム生成 |
| `/generating-spec` で技術設計 |
| `/estimating` で工数見積もり |
| レイアウトの修正・追加が必要であれば指示を受ける |

## 制約

- 要件にない画面を追加しない。一般的に必要な画面（ログイン、エラー等）のみ例外（備考に記載）
- レイアウトはワイヤーフレームレベルの概要に留める（ピクセル詳細やカラー指定は `/design` の領域）
- 仮定を置いた場合は `> ⚠️ 仮定：` で明示

## 関連コマンド

- `/generate-screens` — この入力となる画面一覧・遷移図を生成
- `/generate-diagrams` — FigJam にダイアグラムを生成
- `/generating-spec` — この出力を元に技術設計を行う
- `/define-requirements` — この入力となる要件定義書を生成
