# /extract-requirements — ミーティング・ヒアリングデータから情報を構造化抽出

ミーティングやヒアリングのデータを読み込み、内容を分析して適切な形式で構造化抽出する。要件定義 MTG に限らず、勉強会・社内検討会・クライアント MTG 等あらゆる種類の会議データに対応。

## 引数

- `$ARGUMENTS` — 以下のいずれか:
  - Google Drive フォルダ URL（素材が格納されたフォルダ）
  - Google Docs URL（Gemini メモ等の単体ドキュメント）
  - ローカルファイルパス

## Step 1: 入力ソースの取得

| 入力形式 | 取得方法 |
|---------|---------|
| Google Drive フォルダ URL | `mcp__google-workspace__list_drive_items` → Google Docs は `get_doc_as_markdown`、テキスト/PDF は `get_drive_file_content` で取得 → 全テキストを結合 |
| Google Docs URL | `mcp__google-workspace__get_doc_as_markdown` |
| ローカルファイルパス | Read ツール |

Google Drive アクセスエラー時はローカルファイルでの代替入力を提案する。

## Step 2: コンテンツ種別の自動判定

| 種別 | 判定基準 | 出力 |
|------|---------|------|
| クライアント MTG（要件定義） | クライアントとの対話、課題・要望・システム要件の言及が多い | → Step 3A |
| クライアント MTG（その他） | クライアントとの対話だが要件定義以外（進捗報告、レビュー等） | → Step 3B |
| 勉強会・セミナー | 技術トピックの解説、知見共有、Q&A が中心 | → Step 3C |
| 社内検討・ブレスト | 自社メンバー間の議論、意思決定、方針検討 | → Step 3D |

判定に迷う場合は AskUserQuestion で確認（選択肢: 上記 4 種別）。

---

## Step 3A: 要件定義書（クライアント MTG・要件定義向け）

分析観点: 発言者の分類（クライアント/自社） / 課題の特定 / 要望の特定 / 強調箇所 / 既存システム情報

出力構成:

**ファイルタイトル**: `# {プロジェクト名} 要件定義書`

| セクション | 内容 |
|-----------|------|
| `## 基本情報` | クライアント / ミーティング日 / ソース / 作成日 をテーブルで |
| `## ユーザーの課題` | カラム: `#` / `課題` / `深刻度`（高/中/低） / `現状の対処` / `関連発言`（原文引用） |
| `## ユーザーの要望` | カラム: `#` / `要望` / `優先度`（高/中/低） / `関連する課題`（課題#参照） / `関連発言` |
| `## 機能要件一覧` | カラム: `#` / `機能名` / `概要` / `対応する課題/要望` / `優先度`（高/中/低） / `備考` |
| `## 非機能要件` | カラム: `#` / `カテゴリ` / `要件` / `関連発言` |
| `## 優先度マトリクス` | 最優先（Must Have） / 重要（Should Have） / あれば良い（Nice to Have）の 3 段階で箇条書き |
| `## 前提条件・制約` | 箇条書き |
| `## 未確認事項` | `> ⚠️ 仮定：` の後にチェックリスト形式 |

---

## Step 3B: MTG インサイトレポート（クライアント MTG・その他向け）

分析観点: 議題と結論 / クライアントの反応 / 決定事項 / ネクストアクション / 潜在的な要件

出力構成:

**ファイルタイトル**: `# {プロジェクト/クライアント名} MTG インサイトレポート`

| セクション | 内容 |
|-----------|------|
| `## 基本情報` | クライアント / ミーティング日 / ミーティング種別 / ソース をテーブルで |
| `## 議題と結論` | カラム: `#` / `議題` / `結論・決定事項` / `備考` |
| `## クライアントの反応・温度感` | カラム: `トピック` / `反応`（ポジティブ/ネガティブ/ニュートラル） / `関連発言` |
| `## 決定事項` | チェックリスト形式（担当・期限を括弧内に記載） |
| `## ネクストアクション` | カラム: `#` / `アクション` / `担当` / `期限` / `優先度` |
| `## 潜在的な要件・変更の兆候` | 箇条書き |
| `## 所感・リスク` | 箇条書き |

---

## Step 3C: ナレッジノート（勉強会・セミナー向け）

分析観点: テーマと目的 / 主要な知見 / 実務への応用 / Q&A / 参考リソース

出力構成:

**ファイルタイトル**: `# {テーマ} ナレッジノート`

| セクション | 内容 |
|-----------|------|
| `## 基本情報` | テーマ / 発表者・講師 / 開催日 / ソース をテーブルで |
| `## サマリ（3行で）` | 番号付きリストで 3 つのポイント |
| `## 主要な知見` | トピックごとに `### {トピック名}` で見出し。各トピックにポイント / 詳細 / 関連発言を箇条書き |
| `## 実務への応用` | カラム: `知見` / `適用先` / `具体的なアクション` / `優先度` |
| `## Q&A` | カラム: `質問` / `回答` / `補足` |
| `## 参考リソース` | `- [リソース名](URL) — 概要` の形式 |
| `## 今後深掘りしたい点` | チェックリスト形式 |

---

## Step 3D: 検討レポート（社内検討・ブレスト向け）

分析観点: 検討テーマ / アイデア・意見 / 論点と結論 / 意思決定 / 残課題

出力構成:

**ファイルタイトル**: `# {テーマ} 検討レポート`

| セクション | 内容 |
|-----------|------|
| `## 基本情報` | 検討テーマ / 参加者 / 開催日 / ソース をテーブルで |
| `## 背景・目的` | 自由記述 |
| `## 検討内容` | 論点ごとに `### 論点 N: {タイトル}`。各論点に比較表（`意見・アイデア` / `提案者` / `メリット` / `デメリット`）+ `**結論**: ...` |
| `## 意思決定事項` | カラム: `#` / `決定内容` / `理由` / `影響範囲` |
| `## ネクストアクション` | カラム: `#` / `アクション` / `担当` / `期限` / `優先度` |
| `## 残課題（継続検討）` | チェックリスト形式 |
| `## 所感・リスク` | 箇条書き |

---

## Step 4: 出力ファイルの保存

| 種別 | 出力先 |
|------|--------|
| 要件定義書 | `docs/outputs/{案件slug}/requirements-extracted.md` |
| MTG インサイトレポート | `docs/outputs/{案件slug}/meetings/YYYY-MM-DD-insights.md` |
| ナレッジノート | `docs/outputs/{案件slug}/knowledge/YYYY-MM-DD.md` |
| 検討レポート | `docs/outputs/{案件slug}/discussions/YYYY-MM-DD.md` |

`{案件slug}` はクライアント名・テーマから英語の短いキーワードで生成。同じ案件では同じ slug を使う。

## Step 5: 結果の報告

報告項目: 種別 / 出力先ファイルパス / 抽出結果サマリ（件数等）

次のアクション:

| 種別 | 推奨 |
|------|------|
| 要件定義書 | `/generate-screens` で画面設計、`/generating-spec` で技術設計 |
| MTG インサイトレポート | ネクストアクションのイシュー化を提案 |
| ナレッジノート | 実務適用アクションのイシュー化を提案 |
| 検討レポート | 決定事項の反映、残課題のイシュー化を提案 |

## 制約

- データに明示されていない内容を推測で追加しない。推測が必要な場合は「未確認事項」や「残課題」に記載
- 発言はなるべく原文を引用して「関連発言」に残す（根拠の透明性）
- 要件定義書では機能要件を課題・要望と必ず紐づける（トレーサビリティ）
- 優先度はデータ内の強調度合いから判断。判断が難しい場合は「中」
- Google Drive アクセスエラー時はローカルファイルでの代替入力を提案

## 関連コマンド

- `/generate-screens` — 要件定義書を入力として画面設計
- `/generating-spec` — 要件定義書を入力として技術設計
- `/define-requirements` — より詳細な要件定義が必要な場合に使用
