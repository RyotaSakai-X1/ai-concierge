# /extract-requirements — ミーティング・ヒアリングデータから情報を構造化抽出

ミーティングやヒアリングのデータ（Google Drive 上の Gemini メモ・チャットログ、ローカル議事録等）を読み込み、内容を分析して適切な形式で構造化抽出します。

要件定義ミーティングに限らず、勉強会・社内検討会・クライアントMTGなど、あらゆる種類の会議データに対応します。

## 引数

- `$ARGUMENTS` — 以下のいずれか:
  - Google Drive フォルダ URL（素材が格納されたフォルダ）
  - Google Docs URL（Gemini メモ等の単体ドキュメント）
  - ローカルファイルパス（`docs/outputs/meetings/*.md` 等の議事録）

## 実行手順

### Step 1: 入力ソースの判定と取得

引数の形式に応じて分岐する:

#### パターン A: Google Drive フォルダ URL

1. URL からフォルダ ID を抽出
2. `mcp__google-workspace__list_drive_items` でフォルダ内のファイル一覧を取得
3. 以下のファイルタイプを優先的に取得:
   - **Google Docs**（Gemini メモ等）: `mcp__google-workspace__get_doc_as_markdown` で取得
   - **テキストファイル**（チャットログ等）: `mcp__google-workspace__get_drive_file_content` で取得
   - **PDF**: `mcp__google-workspace__get_drive_file_content` で取得
4. 取得した全テキストを結合して分析対象とする

#### パターン B: Google Docs URL

1. URL からドキュメント ID を抽出
2. `mcp__google-workspace__get_doc_as_markdown` で内容を取得

#### パターン C: ローカルファイルパス

1. Read ツールでファイル内容を取得
2. 複数ファイルが指定された場合は全て読み込む

### Step 2: コンテンツ種別の自動判定

取得したテキスト全体を読み、以下の種別のうちどれに該当するか判定する:

| 種別                           | 判定基準                                                   | 出力形式                       |
| ------------------------------ | ---------------------------------------------------------- | ------------------------------ |
| **クライアントMTG（要件定義）** | クライアントとの対話、課題・要望・システム要件の言及が多い   | → Step 3A: 要件定義書           |
| **クライアントMTG（その他）**   | クライアントとの対話だが、要件定義以外（進捗報告、レビュー等） | → Step 3B: MTGインサイトレポート |
| **勉強会・セミナー**           | 技術トピックの解説、知見共有、Q&A が中心                    | → Step 3C: ナレッジノート       |
| **社内検討・ブレスト**         | 自社メンバー間の議論、意思決定、方針検討                    | → Step 3D: 検討レポート         |

判定に迷う場合は AskUserQuestion でユーザーに確認する:
- 選択肢: `クライアントMTG（要件定義）` / `クライアントMTG（その他）` / `勉強会・セミナー` / `社内検討・ブレスト`

---

### Step 3A: 要件定義書（クライアントMTG・要件定義向け）

#### 分析観点

1. **発言者の分類**: クライアント側 / 自社側の発言を区別する
2. **課題の特定**: クライアントが「困っている」「課題がある」と述べている箇所を抽出
3. **要望の特定**: クライアントが「こうしたい」「欲しい」と述べている箇所を抽出
4. **強調箇所の特定**: 繰り返し言及されている内容を優先度高とマーク
5. **既存システム情報**: 現在使っているツール・システムの情報を収集

#### 出力フォーマット

以下の構造に従って Markdown ファイルを生成する。見出し `#` はファイルのトップレベルとして使用する。

**ファイルタイトル**: `# {プロジェクト名} 要件定義書`

**基本情報セクション** (`## 基本情報`):

| 項目           | 記載内容               |
| -------------- | ---------------------- |
| クライアント   | クライアント名         |
| ミーティング日 | YYYY-MM-DD             |
| ソース         | 入力元のファイル/URL   |
| 作成日         | 今日の日付             |

**ユーザーの課題セクション** (`## ユーザーの課題`):

各課題を以下のカラムで表にする: `#` / `課題` / `深刻度`（高/中/低） / `現状の対処` / `関連発言`（原文引用）

**ユーザーの要望セクション** (`## ユーザーの要望`):

各要望を以下のカラムで表にする: `#` / `要望` / `優先度`（高/中/低） / `関連する課題`（課題#の参照） / `関連発言`（原文引用）

**機能要件一覧セクション** (`## 機能要件一覧`):

各機能を以下のカラムで表にする: `#` / `機能名` / `概要` / `対応する課題/要望`（課題#・要望#の参照） / `優先度`（高/中/低） / `備考`

**非機能要件セクション** (`## 非機能要件`):

各要件を以下のカラムで表にする: `#` / `カテゴリ`（パフォーマンス/セキュリティ等） / `要件` / `関連発言`

**優先度マトリクスセクション** (`## 優先度マトリクス`):

3段階で機能を分類する:
- `### 最優先（Must Have）` — 箇条書きで機能#と概要
- `### 重要（Should Have）` — 箇条書きで機能#と概要
- `### あれば良い（Nice to Have）` — 箇条書きで機能#と概要

**前提条件・制約セクション** (`## 前提条件・制約`): 箇条書き

**未確認事項セクション** (`## 未確認事項`): `> ⚠️ 仮定：` の引用ブロックの後にチェックリスト形式で記載

---

### Step 3B: MTGインサイトレポート（クライアントMTG・その他向け）

#### 分析観点

1. **議題と結論**: 何が議論され、どう結論が出たか
2. **クライアントの反応**: ポジティブ/ネガティブな反応、懸念点
3. **決定事項**: 合意された内容
4. **ネクストアクション**: 次にやるべきこと、担当者、期限
5. **潜在的な要件**: MTG中に暗示的に示された要件や変更の兆候

#### 出力フォーマット

**ファイルタイトル**: `# {プロジェクト/クライアント名} MTGインサイトレポート`

**基本情報セクション** (`## 基本情報`):

| 項目             | 記載内容                       |
| ---------------- | ------------------------------ |
| クライアント     | クライアント名                 |
| ミーティング日   | YYYY-MM-DD                     |
| ミーティング種別 | 進捗報告 / レビュー / 相談 等 |
| ソース           | 入力元                         |

**議題と結論セクション** (`## 議題と結論`):

各議題を以下のカラムで表にする: `#` / `議題` / `結論・決定事項` / `備考`

**クライアントの反応・温度感セクション** (`## クライアントの反応・温度感`):

各トピックを以下のカラムで表にする: `トピック` / `反応`（ポジティブ/ネガティブ/ニュートラル） / `関連発言`（原文引用）

**決定事項セクション** (`## 決定事項`): チェックリスト形式（担当・期限を括弧内に記載）

**ネクストアクションセクション** (`## ネクストアクション`):

各アクションを以下のカラムで表にする: `#` / `アクション` / `担当` / `期限` / `優先度`（高/中/低）

**潜在的な要件・変更の兆候セクション** (`## 潜在的な要件・変更の兆候`): 箇条書き

**所感・リスクセクション** (`## 所感・リスク`): 箇条書き

---

### Step 3C: ナレッジノート（勉強会・セミナー向け）

#### 分析観点

1. **テーマと目的**: 何についての勉強会か
2. **主要な知見**: 重要なポイント、新しい学び
3. **実務への応用**: 自社プロジェクトに活かせる点
4. **Q&A**: 質疑応答の内容
5. **参考リソース**: 言及されたツール、ライブラリ、記事

#### 出力フォーマット

**ファイルタイトル**: `# {テーマ} ナレッジノート`

**基本情報セクション** (`## 基本情報`):

| 項目       | 記載内容         |
| ---------- | ---------------- |
| テーマ     | 勉強会のテーマ   |
| 発表者/講師 | 名前             |
| 開催日     | YYYY-MM-DD       |
| ソース     | 入力元           |

**サマリセクション** (`## サマリ（3行で）`): 番号付きリストで3つのポイント

**主要な知見セクション** (`## 主要な知見`):

トピックごとに `### {トピック名}` で見出しを作り、各トピックに以下を箇条書き:
- **ポイント**: 一言でまとめ
- **詳細**: 具体的な説明
- **関連発言**: 原文引用

**実務への応用セクション** (`## 実務への応用`):

各項目を以下のカラムで表にする: `知見` / `適用先` / `具体的なアクション` / `優先度`（高/中/低）

**Q&Aセクション** (`## Q&A`):

各項目を以下のカラムで表にする: `質問` / `回答` / `補足`

**参考リソースセクション** (`## 参考リソース`): `- [リソース名](URL) — 概要` の形式

**今後深掘りしたい点セクション** (`## 今後深掘りしたい点`): チェックリスト形式

---

### Step 3D: 検討レポート（社内検討・ブレスト向け）

#### 分析観点

1. **検討テーマ**: 何を検討したか
2. **出たアイデア・意見**: 参加者から出た意見を網羅的に
3. **論点と結論**: 議論が分かれた点、最終的な方向性
4. **意思決定**: 決まったこと
5. **残課題**: 未決定・継続検討の事項

#### 出力フォーマット

**ファイルタイトル**: `# {テーマ} 検討レポート`

**基本情報セクション** (`## 基本情報`):

| 項目       | 記載内容     |
| ---------- | ------------ |
| 検討テーマ | テーマ       |
| 参加者     | メンバー一覧 |
| 開催日     | YYYY-MM-DD   |
| ソース     | 入力元       |

**背景・目的セクション** (`## 背景・目的`): 自由記述

**検討内容セクション** (`## 検討内容`):

論点ごとに `### 論点 N: {論点タイトル}` で見出しを作り、各論点に以下を含む:
- 意見・アイデアの比較表: `意見・アイデア` / `提案者` / `メリット` / `デメリット` のカラム
- `**結論**: ...` で結論を明記

**意思決定事項セクション** (`## 意思決定事項`):

各決定を以下のカラムで表にする: `#` / `決定内容` / `理由` / `影響範囲`

**ネクストアクションセクション** (`## ネクストアクション`):

各アクションを以下のカラムで表にする: `#` / `アクション` / `担当` / `期限` / `優先度`（高/中/低）

**残課題セクション** (`## 残課題（継続検討）`): チェックリスト形式

**所感・リスクセクション** (`## 所感・リスク`): 箇条書き

---

### Step 4: 出力ファイルの保存

種別に応じて出力先ディレクトリを決定する:

| 種別                 | 出力先                                                 |
| -------------------- | ------------------------------------------------------ |
| 要件定義書           | `docs/outputs/requirements/YYYY-MM-DD-{slug}.md`       |
| MTGインサイトレポート | `docs/outputs/meetings/YYYY-MM-DD-{slug}-insights.md`  |
| ナレッジノート       | `docs/outputs/knowledge/YYYY-MM-DD-{slug}.md`          |
| 検討レポート         | `docs/outputs/discussions/YYYY-MM-DD-{slug}.md`        |

- `YYYY-MM-DD` は今日の日付、`{slug}` はテーマ・クライアント名から生成（英数字・ハイフン）
- ディレクトリが存在しない場合は作成する

### Step 5: 結果の報告

種別に応じた報告を行う:

- **種別**: 判定された種別
- **出力先**: ファイルパス
- **抽出結果**: 種別に応じたサマリ（件数等）

推奨する次のアクション:

| 種別                 | 推奨アクション                                                     |
| -------------------- | ------------------------------------------------------------------ |
| 要件定義書           | `/generate-screens` で画面設計、`/generating-spec` で技術設計       |
| MTGインサイトレポート | ネクストアクションのイシュー化を提案                                |
| ナレッジノート       | 実務適用アクションのイシュー化を提案                                |
| 検討レポート         | 決定事項の反映、残課題のイシュー化を提案                            |

## 注意事項

- データに明示されていない内容を「推測」で追加しない。推測が必要な場合は「未確認事項」や「残課題」に記載する
- 発言はなるべく原文を引用して「関連発言」に残す（根拠の透明性）
- 要件定義書では機能要件を課題・要望と必ず紐づける（トレーサビリティ）
- 優先度はデータ内の強調度合いから判断する。判断が難しい場合は「中」とする
- Google Drive アクセスでエラーが発生した場合は、ローカルファイルでの代替入力を提案する

## 関連コマンド

- `/generate-screens` — 要件定義書を入力として画面設計を行う
- `/generating-spec` — 要件定義書を入力として技術設計を行う
- `/define-requirements` — より詳細な要件定義が必要な場合に使用
