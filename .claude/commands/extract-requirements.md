# /extract-requirements — ヒアリングデータから要件を構造化抽出

ヒアリングデータ（Google Drive 上の Gemini メモ・チャットログ、またはローカルの議事録ファイル）を読み込み、ユーザーの課題と機能要件を構造化して抽出します。

## 引数

- `$ARGUMENTS` — 以下のいずれか:
  - Google Drive フォルダ URL（ヒアリング素材が格納されたフォルダ）
  - Google Docs URL（Gemini メモ等の単体ドキュメント）
  - ローカルファイルパス（`docs/outputs/meetings/*.md` 等の議事録）

## 実行手順

### Step 1: 入力ソースの判定と取得

引数の形式に応じて分岐する:

#### パターン A: Google Drive フォルダ URL

1. URL からフォルダ ID を抽出
2. `mcp__google-workspace__list_drive_items` でフォルダ内のファイル一覧を取得
3. 以下のファイルタイプを優先的に取得:
   - **Google Docs**（Gemini メモ等）: `mcp__google-workspace__get_doc_as_markdown` で取得
   - **テキストファイル**（チャットログ等）: `mcp__google-workspace__get_drive_file_content` で取得
   - **PDF**: `mcp__google-workspace__get_drive_file_content` で取得
4. 取得した全テキストを結合して分析対象とする

#### パターン B: Google Docs URL

1. URL からドキュメント ID を抽出
2. `mcp__google-workspace__get_doc_as_markdown` で内容を取得

#### パターン C: ローカルファイルパス

1. Read ツールでファイル内容を取得
2. 複数ファイルが指定された場合は全て読み込む

### Step 2: ヒアリングデータの分析

取得したテキストを以下の観点で分析する:

1. **発言者の分類**: クライアント側 / 自社側の発言を区別する
2. **課題の特定**: クライアントが「困っている」「課題がある」と述べている箇所を抽出
3. **要望の特定**: クライアントが「こうしたい」「欲しい」と述べている箇所を抽出
4. **強調箇所の特定**: 繰り返し言及されている内容、強い口調で述べられている箇所を優先度高とマーク
5. **既存システム情報**: 現在使っているツール・システムの情報を収集

### Step 3: 要件の構造化

分析結果を以下の構造に整理する:

```markdown
# {プロジェクト名} 要件定義書

## 基本情報

| 項目 | 内容 |
|------|------|
| クライアント | {クライアント名} |
| ヒアリング日 | {日付} |
| ソース | {入力元のファイル/URL} |
| 作成日 | {今日の日付} |

## ユーザーの課題

| # | 課題 | 深刻度 | 現状の対処 | 関連発言 |
|---|------|--------|-----------|---------|
| 1 | ... | 高/中/低 | ... | 「...」 |

## ユーザーの要望

| # | 要望 | 優先度 | 関連する課題 | 関連発言 |
|---|------|--------|-------------|---------|
| 1 | ... | 高/中/低 | #1, #2 | 「...」 |

## 機能要件一覧

| # | 機能名 | 概要 | 対応する課題/要望 | 優先度 | 備考 |
|---|--------|------|------------------|--------|------|
| 1 | ... | ... | 課題#1, 要望#1 | 高/中/低 | ... |

## 非機能要件

| # | カテゴリ | 要件 | 関連発言 |
|---|---------|------|---------|
| 1 | パフォーマンス | ... | 「...」 |
| 2 | セキュリティ | ... | 「...」 |

## 優先度マトリクス

### 最優先（Must Have）
- 機能#X: ...
- 機能#Y: ...

### 重要（Should Have）
- 機能#Z: ...

### あれば良い（Nice to Have）
- 機能#W: ...

## 前提条件・制約

- ...

## 未確認事項

> ⚠️ 仮定：以下の点はヒアリングデータから明確に読み取れなかったため、確認が必要です。

- [ ] ...
```

### Step 4: 出力ファイルの保存

1. 出力先ディレクトリを決定:
   ```
   docs/outputs/requirements/YYYY-MM-DD-{client-slug}.md
   ```
   - `YYYY-MM-DD` は今日の日付
   - `{client-slug}` はクライアント名やプロジェクト名から生成（英数字・ハイフン）
2. ディレクトリが存在しない場合は作成する
3. ファイルを書き出す

### Step 5: 結果の報告

以下の形式でユーザーに報告する:

```
## 要件抽出完了

- 📄 出力先: `docs/outputs/requirements/YYYY-MM-DD-{slug}.md`
- 📊 抽出結果:
  - 課題: X 件
  - 要望: X 件
  - 機能要件: X 件（Must: X / Should: X / Nice: X）
  - 非機能要件: X 件
  - 未確認事項: X 件

### 次のアクション
- `/generate-screens` で画面設計に進む
- `/generating-spec` で技術設計に進む
```

## 注意事項

- ヒアリングデータに明示されていない要件を「推測」で追加しない。推測が必要な場合は「未確認事項」に記載する
- クライアントの発言はなるべく原文を引用して「関連発言」に残す（根拠の透明性）
- 機能要件は課題・要望と必ず紐づける（トレーサビリティ）
- 優先度はヒアリング内の強調度合いから判断する。判断が難しい場合は「中」とし、未確認事項に追記する
- Google Drive アクセスでエラーが発生した場合は、ローカルファイルでの代替入力を提案する

## 関連コマンド

- `/generate-screens` — この出力を入力として画面設計を行う
- `/generating-spec` — この出力を入力として技術設計を行う
- `/define-requirements` — より詳細な要件定義が必要な場合に使用
