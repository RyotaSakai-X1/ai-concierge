# ロードマップ

## 最終ゴール：Agent SDK によるエージェントチーム

このプロジェクトの最終目標は、**複数の AI エージェントが役割分担して自律的に並行処理する仕組み**を作ること。
会社組織のように、PM・レビュアー・実行者がそれぞれ同時に動く状態を目指す。

```
オーケストレーター（司令塔）
  ├── PM エージェント: ideas.md を監視 → 仕様書を自動生成
  ├── レビューエージェント: 仕様書・成果物を自動レビュー
  ├── 実行エージェント: 承認済みタスクを自動実行
  └── 報告エージェント: 日次レポート・進捗まとめを生成
```

### ゴールに至るまでの3フェーズ

| フェーズ | やること | 状態 |
|---------|---------|------|
| Phase 1 | コマンドとナレッジを手動で回し、精度を上げる | 🔄 今ここ |
| Phase 2 | GitHub Actions で一部を自動トリガー化 | 未着手 |
| Phase 3 | Agent SDK でエージェントチームを構築 | 未着手 |

## 優先度：高

### 1. hooks（自動ガードレール）の導入

**何ができるようになる？**
「人が忘れても仕組みが自動で守ってくれる」安全装置。

**置き場所:** `.claude/hooks/`

**導入したい hooks の例:**

| hook | タイミング | やること |
|------|----------|---------|
| 機密情報チェック | コミット前 | `.env` や API キーがコミットに含まれていないか自動検出 |
| PR 自動作成 | `outputs/` に成果物保存後 | 成果物を保存したら自動で PR を作成 |
| 変更確認アラート | `knowledge/` 変更時 | ルール系ファイルの変更時に「本当に変えていい？」と確認 |
| フォーマッター | ファイル編集後 | Markdown の書式を自動整形 |

**なぜ優先度が高い？**
非エンジニアが使うプロジェクトでは、ルールを覚えてもらうより仕組みで守るほうが確実。

## 優先度：中

### 2. ローカル CLAUDE.md（フォルダごとの注意書き）

**何ができるようになる？**
Claude がそのフォルダで作業するとき、自動でそのフォルダ固有の注意点を読み込む。

**導入イメージ:**

| 場所 | 書く内容 |
|------|---------|
| `knowledge/estimation/CLAUDE.md` | 見積もり基準を変更する場合は過去実績との整合性を必ず確認すること |
| `outputs/CLAUDE.md` | ここに直接ファイルを作らず、必ずコマンド経由で出力すること |
| `templates/CLAUDE.md` | テンプレートを変更したら、対応するコマンド定義も更新すること |

**なぜ有効？**
フォルダが増えてきたときに、CLAUDE.md 本体を肥大化させずに済む。必要なときだけ必要な知識が読み込まれる。

### 3. docs/ の構造化

**何ができるようになる？**
「なぜそうしたか」の意思決定と「どうやるか」の手順書を整理できる。

**導入イメージ:**

```
docs/
├── decisions/          ← 意思決定の記録（ADR）
│   ├── 001_見積もりコマンドの設計.md
│   └── 002_ディレクトリ構成の方針.md
├── runbooks/           ← 操作手順書（非エンジニア向け）
│   ├── claude-code-setup.md    ← Claude Code の初期設定
│   └── how-to-use-commands.md  ← コマンドの使い方ガイド
└── roadmap.md          ← このファイル
```

**decisions/（意思決定記録）が便利な理由:**
- 後から「なんでこうなってるの？」と聞かれたときの回答になる
- Claude もこれを読むことで、過去の判断を踏まえた提案ができる

**runbooks/（手順書）が便利な理由:**
- 非エンジニアが迷わず使い始められる
- 「セットアップどうやるの？」を毎回聞かなくて済む
