# ロードマップ

## 最終ゴール：Agent tool + worktree 並列実行による高効率エンジニアリング

このプロジェクトの最終目標は、**Claude Code CLI をテックリードとして運用し、エンジニアリング・デザイン業務を高効率に回す仕組み**を作ること。
Agent tool の `isolation: "worktree"` による並列実行で、複数イシューを同時処理できる状態を目指す。

```
Claude Code CLI（テックリードエージェント）
  ├── [コーディング系]
  │     ├── /work-on-issue → 単体イシューの実装〜PR
  │     ├── /parallel-work → 複数イシューの並列実行（worktree 分離）
  │     ├── /reviewing     → セルフレビュー・PRレビュー
  │     └── /creating-pr   → 変更からブランチ作成〜PR作成
  ├── [分析系]
  │     ├── /extract-requirements → ヒアリングデータから要件抽出
  │     ├── /estimating           → 工数見積もり
  │     └── /generate-screens     → 画面一覧・画面遷移図の生成
  ├── [テスター系]
  │     ├── /generate-test-cases → コードベース+仕様書からテスト項目を抽出しスプシに書き出し
  │     ├── /run-ui-test         → テスト項目に基づくUIテスト実行・結果記録
  │     └── /test-report         → テスト結果集計・NG項目からバグイシュー自動起票
  ├── [デザイナー系]
  │     ├── /sync-design-system → Figma からデザイントークン抽出・保存
  │     ├── /design             → デザインシステム準拠の UI コード生成
  │     └── /design-review      → UI コードのデザインシステム適合性レビュー
  └── [横断系]
        ├── /briefing        → エンジニアリングブリーフィング
        ├── /generating-spec → 仕様書から実装計画・技術設計を生成
        ├── /executing-plan  → タスクリストを順次自律実行
        └── /checking-ready  → イシュー着手可否チェック
```

### ゴールに至るまでの3フェーズ

| フェーズ | やること | 状態 |
|---------|---------|------|
| Phase 1 | コマンド・ルール・ナレッジを手動運用で磨き込む | 🔄 今ここ |
| Phase 2 | hooks と GitHub Actions で自動化を導入 | 未着手 |
| Phase 3 | 並列実行の最適化・ナレッジの体系的公開 | 未着手 |

## Phase 1：基盤固め（今ここ）

コマンドとルールの精度を上げ、日常業務で安定して使えるようにするフェーズ。

参考: [The Anatomy of a Claude Code Project](https://x.com/Suryanshti777/status/2030518942087995541)

### 優先度：高

#### 1. hooks（自動ガードレール）の導入

**何ができるようになる？**
「人が忘れても仕組みが自動で守ってくれる」安全装置。

**置き場所:** `.claude/hooks/`

| hook | タイミング | やること |
|------|----------|---------|
| 機密情報チェック | コミット前 | `.env` や API キーがコミットに含まれていないか自動検出 |
| PR 自動作成 | `outputs/` に成果物保存後 | 成果物を保存したら自動で PR を作成 |
| 変更確認アラート | `knowledge/` 変更時 | ルール系ファイルの変更時に確認 |
| フォーマッター | ファイル編集後 | Markdown の書式を自動整形 |

---

### 優先度：中

#### 2. ローカル CLAUDE.md（フォルダごとの注意書き）

フォルダ固有の注意点を自動で読み込ませる仕組み。CLAUDE.md 本体の肥大化を防ぐ。

| 場所 | 書く内容 |
|------|---------|
| `outputs/CLAUDE.md` | ここに直接ファイルを作らず、必ずコマンド経由で出力すること |
| `templates/CLAUDE.md` | テンプレートを変更したら、対応するコマンド定義も更新すること |

#### 3. docs/ の構造化

```
docs/
├── decisions/          ← 意思決定の記録（ADR）
├── runbooks/           ← 操作手順書
└── roadmap.md          ← このファイル
```

#### 4. イシュー起点のブランチ運用

main への直コミットを廃止し、すべての作業をイシュー起点で行う。
`/work-on-issue` → ブランチ自動作成 → 作業 → `/creating-pr` → レビュー → マージ

#### 5. 分析系コマンドの整備

コーディング系とは別レーンで、要件定義・見積もり・画面設計のコマンドを整備する。

| コマンド | 用途 | イシュー |
|---------|------|---------|
| `/extract-requirements` | ヒアリングデータから要件抽出 | #36 |
| `/estimating` | 工数見積もり | 整備済み |
| `/generate-screens` | 画面一覧・画面遷移図の生成 | #37 |

#### 6. テスター系コマンドの整備

テスト管理をスプレッドシート連携で効率化する。現段階は手動実行+記録、将来的に Playwright MCP で自動化を拡張。

| コマンド | 用途 | 備考 |
|---------|------|------|
| `/generate-test-cases` | コードベース+仕様書からテスト項目を抽出しスプシに書き出し | Google Workspace MCP 連携 |
| `/run-ui-test` | テスト項目に基づくUIテスト実行・結果記録 | Browser Use CLI 2.0 で自動化（自然言語操作、Real Browser モード対応） |
| `/test-report` | テスト結果集計・NG項目からバグイシュー自動起票 | GitHub Issues 連携 |
