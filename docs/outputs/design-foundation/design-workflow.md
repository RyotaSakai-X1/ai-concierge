# AI デザイナー ワークフロー定義

## 概要

Figma MCP を活用した AI デザイナーの意思決定フローを定義する。
AIが自動化できる部分と、人間の承認が必要なポイントを明確にする。

## ワークフロー全体像

```
入力 → 調査・分析 → 設計判断 → 出力生成 → 人間レビュー → 反映
```

### フェーズ別フロー

#### Phase A: デザインシステム構築（/sync-design-system）

入力: Figma ファイル URL
出力: デザイントークン定義、コンポーネントカタログ

| ステップ | AI が実行 | 人間の承認 | 使用ツール |
|---------|----------|-----------|-----------|
| 1. ファイル構造の取得 | get_metadata でページ・セクション・コンポーネント一覧を取得 | 不要 | get_metadata |
| 2. デザイントークン抽出 | get_variable_defs でカラー・タイポグラフィ・スペーシングを取得 | 不要 | get_variable_defs |
| 3. スクリーンショット取得 | 主要コンポーネントのスクリーンショットを取得 | 不要 | get_screenshot |
| 4. トークン整理・命名 | 抽出したトークンを整理し、命名規則を提案 | **必要** — トークン名の承認 | — |
| 5. コンポーネントカタログ生成 | コンポーネント一覧とバリエーションをドキュメント化 | **必要** — 一覧の確認 | — |
| 6. Code Connect 設定 | Figma コンポーネントとコードの紐づけを提案 | **必要** — マッピングの承認 | get_code_connect_suggestions, send_code_connect_mappings |

#### Phase B: 画面デザイン（/design）

入力: 要件定義書 or ワイヤーフレーム
出力: Figma デザインファイル

| ステップ | AI が実行 | 人間の承認 | 使用ツール |
|---------|----------|-----------|-----------|
| 1. 要件の解釈 | 要件定義書からUI要素・レイアウト方針を抽出 | 不要 | — |
| 2. ワイヤーフレーム参照 | FigJam のワイヤーフレームからレイアウト情報を取得 | 不要 | get_figjam |
| 3. デザイントークン確認 | 既存デザインシステムのトークンを確認 | 不要 | get_variable_defs |
| 4. レイアウト設計 | コンポーネントの配置・サイズ・間隔を決定 | **必要** — レイアウト方針の承認 | — |
| 5. ダイアグラム生成 | 画面遷移図やフローチャートを FigJam に生成 | 不要 | generate_diagram |
| 6. 参考デザイン参照 | 既存画面のデザインコンテキストを取得 | 不要 | get_design_context |
| 7. 実装コード生成 | 承認済みデザインからコードを生成 | **必要** — コードの確認 | get_design_context |

#### Phase C: デザインレビュー（/design-review）

入力: Figma デザインファイル URL
出力: レビューレポート

| ステップ | AI が実行 | 人間の承認 | 使用ツール |
|---------|----------|-----------|-----------|
| 1. デザイン取得 | 対象ノードのスクリーンショットとメタデータを取得 | 不要 | get_screenshot, get_metadata |
| 2. デザインコンテキスト取得 | コード生成用のデザイン情報を取得 | 不要 | get_design_context |
| 3. トークン整合性チェック | 使用されているトークンがデザインシステムに準拠しているか検証 | 不要 | get_variable_defs |
| 4. Code Connect チェック | コンポーネントが正しくマッピングされているか確認 | 不要 | get_code_connect_map |
| 5. レビューレポート生成 | 問題点・改善提案をドキュメント化 | **必要** — レビュー結果の確認 | — |
| 6. 修正指示 | 承認された修正をデザイナーに伝える | **必要** — 修正内容の承認 | — |

## 自動化可能な領域

| 領域 | 自動化レベル | 説明 |
|------|------------|------|
| デザイントークン抽出 | 完全自動 | get_variable_defs で Figma から直接取得可能 |
| ファイル構造分析 | 完全自動 | get_metadata でノードツリーを取得可能 |
| スクリーンショット取得 | 完全自動 | get_screenshot で任意のノードを画像化 |
| 参考コード生成 | 完全自動 | get_design_context で React+Tailwind コードを生成 |
| ダイアグラム生成 | 完全自動 | generate_diagram で FigJam に図を自動生成 |
| デザイン-コード紐づけ提案 | 半自動 | get_code_connect_suggestions で提案 → 人間が承認 |
| Web → Figma キャプチャ | 半自動 | generate_figma_design でキャプチャ → 出力先を人間が選択 |
| デザイントークン命名 | 半自動 | AI が提案 → 人間が承認 |
| コンポーネント設計 | 半自動 | AI が提案 → 人間が承認・修正 |

## 人間の承認が必要なポイント

| 承認ポイント | 理由 | タイミング |
|-------------|------|-----------|
| デザイントークンの命名規則 | プロジェクト固有の命名方針が必要 | デザインシステム構築時 |
| コンポーネントカタログの構成 | ビジネスドメインの理解が必要 | デザインシステム構築時 |
| レイアウト方針 | UX判断・ブランド方針の反映 | 画面デザイン時 |
| Code Connect マッピング | コードベースの構造判断が必要 | 紐づけ設定時 |
| レビュー結果の確認 | 修正要否の最終判断 | デザインレビュー時 |
| 生成コードの採用 | 品質・パフォーマンス・保守性の判断 | 実装時 |

## 意思決定フロー図

以下の判断ロジックで AI は行動する:

1. **デザインファイルが提供された場合**
   → get_metadata で構造確認 → get_design_context でコンテキスト取得 → 目的に応じた処理

2. **コード実装が必要な場合**
   → get_design_context でコード取得 → プロジェクトのスタックに変換 → 人間に確認

3. **デザインシステム整備が必要な場合**
   → get_variable_defs でトークン抽出 → create_design_system_rules でルールテンプレ取得 → 整理して提案

4. **ワイヤーフレーム → デザインの場合**
   → get_figjam でワイヤー取得 → 既存デザインを get_design_context で参照 → レイアウト提案

## Figma MCP の制約による設計上の注意

| 制約 | 影響 | 対策 |
|------|------|------|
| Figma上のデザイン直接編集不可 | AI がデザインを直接修正できない | 修正指示をドキュメントで出力し、人間デザイナーが反映 |
| コード出力が React+Tailwind 固定 | プロジェクトのスタックと異なる可能性 | AI が自動変換するレイヤーを設ける |
| アセット URL が7日間限定 | 長期保存には不向き | 必要なアセットはプロジェクトにダウンロード保存 |
| プロトタイプ情報取得不可 | インタラクション仕様は AI が読み取れない | インタラクション仕様は別途ドキュメントで管理 |
