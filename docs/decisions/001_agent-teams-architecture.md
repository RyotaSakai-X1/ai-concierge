# ADR-001: Agent Teams + Worktree アーキテクチャの採用

## ステータス: 承認済み
## 日付: 2026-03-12

---

## コンテキスト

ai-concierge は当初「秘書（単一エージェント）がユーザーの指示を受けて逐次的に作業する」設計だった。
Phase 1 でコマンド・ルール・テンプレートの基盤が整い、次のスケーリングを検討する段階に入った。

同時に、以下の外部要因があった：

- **Claude Code の Agent Teams + Worktree 機能** が利用可能になり、複数エージェントの並列実行が現実的になった
- **Cowork（Claude のチーム向けワークスペース）** の登場により、非エンジニア向けの機能は専用ツールで代替可能になった
- 現状の CLAUDE.md・ルール・コマンドがすべて単一エージェント前提で書かれており、並列実行すると競合・重複が発生する

## 決定

### 1. ツールによる領域分割

| 領域 | ツール | 理由 |
|------|--------|------|
| エンジニアリング | このリポジトリ（Claude Code） | Git/GitHub API が必要。ターミナル操作が前提 |
| PM系 | このリポジトリ（一旦残す） | GitHub Issue 操作が必要。後で再判断 |
| ブリーフィング・議事録 | Cowork | Google Calendar/Drive 連携は Cowork の方が適切 |

**根拠：** 部署別に CLAUDE.md を分けるよりも、ツール自体を分ける方が境界が明確でメンテしやすい。

### 2. Agent Teams 組織構造の採用

```
ユーザー → オーケストレーター → {エンジニア, テスター, デザイナー} リーダー → ワーカー(worktree)
```

**根拠：**
- リーダー/ワーカーの2層構造により、技術方針の一貫性を保ちつつ並列実行できる
- 各ワーカーは独立した worktree で作業するため、ファイル競合が原理的に発生しない
- Issue 単位でワーカーに割り当てることで、タスクの重複を防止できる

### 3. 構造変更の段階的実施

Phase 1.5（構造整理）→ Phase 2（並列基盤）→ Phase 3（スケーリング）の3段階で進める。

**根拠：** 一度に全部変えるとリスクが高い。まず構造を整理し、次に2-3エージェントで検証し、安定したらスケールする。

## 影響

### 変更が必要なもの

- **CLAUDE.md** — 単一エージェント前提の記述を並列対応に修正
- **git-workflow.md** — ブランチ命名、main帰還フロー、worktreeクリーンアップ
- **executing-plan** — タスクロック機構の導入
- **creating-pr** — ブランチ名衝突防止
- **secretary-behavior.md** — セッション管理の並列対応

### 新規作成するもの

- `.claude/agents/` — 各ロールの振る舞い定義
- `knowledge/rules/parallel-execution.md` — 並列実行ルール
- `.claude/status/agents.json` — エージェント間状態共有

### 削除/移行するもの

- `/briefing` コマンド → Cowork
- `/meeting-summary` コマンド → Cowork

## 代替案（不採用）

### A. 部署別 CLAUDE.md

CLAUDE.md を `profiles/engineering.md`, `profiles/pm.md` に分割する案。
**不採用理由：** Claude Code は CLAUDE.md を1つしか読まない。コマンド冒頭で読み込ませる運用は可能だが、ツール自体を分ける方がシンプル。

### B. GitHub Actions による自動トリガー（Phase 2 旧案）

ideas.md 更新 → 自動でコマンド実行のパイプライン。
**不採用理由：** Agent Teams + Worktree の方がリアルタイム性・柔軟性が高い。GitHub Actions は補助的に使う。

### C. 単一エージェントのまま高速化

コマンドの実行速度を上げて、逐次でも十分な速度を出す案。
**不採用理由：** Issue の数が増えるとスケールしない。並列化は本質的な改善。
