# Team Design Test - プロジェクト規約

## このプロジェクトは何？

チーム全員が Claude Code（AI アシスタント）を使って、日々の業務を自動化・効率化するためのプロジェクトです。
エンジニアでなくても、チャットで指示するだけで見積もり・報告書作成・議事録整理などができるようになります。

## フォルダの構成

```
.
├── CLAUDE.md                     ← このファイル（プロジェクトのルールブック）
│
├── .claude/commands/             ← 「/○○」で使えるコマンドの定義
│   ├── estimate.md               　 例：/estimate で見積もり自動作成
│   └── README.md                 　 コマンドの一覧表
│
├── knowledge/                    ← 判断の元になるルール・ノウハウ集
│   ├── estimation/               　 例：見積もりの基準や過去実績
│   │   ├── guidelines.md         　 　　見積もりルール
│   │   └── examples/             　 　　過去の見積もり事例
│   └── README.md                 　 ナレッジの一覧表
│
├── templates/                    ← 成果物のひな形
│   └── estimate-report/          　 例：見積もり報告書のフォーマット
│       └── template.md
│
├── outputs/                      ← コマンドで作った成果物の保存先
│   └── estimation/               　 例：各案件の見積もり報告書
│       └── 2026-03-09_○○案件/
│           └── report.md
│
└── docs/                         ← プロジェクト自体の説明書
    └── onboarding.md             　 新メンバー向けガイド
```

### フォルダの役割まとめ

| フォルダ | ひとことで言うと | 誰が使う？ |
|---------|----------------|-----------|
| `.claude/commands/` | コマンドの設計図 | 主にエンジニア（作る人） |
| `knowledge/` | AI に教える知識 | 全員（ノウハウを追記できる） |
| `templates/` | 成果物のひな形 | 全員（フォーマットを調整できる） |
| `outputs/` | 完成した成果物 | 全員（ここに出力される） |
| `docs/` | このプロジェクトの説明書 | 全員 |

## コマンドの使い方

Claude Code のチャットで `/{コマンド名}` と入力するだけです。

```
例：見積もりを作りたい場合
/estimate https://docs.google.com/spreadsheets/d/xxxxx/edit
```

使えるコマンドの一覧は `.claude/commands/README.md` を見てください。

## 新しいコマンドを追加するときのルール

「こんな自動化がしたい」と依頼されたら、以下の手順で対応すること。

### ステップ 1：要件を整理する

まず以下を明確にする：
- **何をしたいか**（例：見積もり報告書を自動で作りたい）
- **何を入力するか**（例：スプレッドシートの URL）
- **何が出力されるか**（例：報告書の Markdown ファイル + PR）

### ステップ 2：必要なファイルを作成する

| 作るもの | 置く場所 | 必須？ |
|---------|---------|-------|
| コマンド定義 | `.claude/commands/{コマンド名}.md` | 必須 |
| 判断基準・ルール | `knowledge/{分野}/guidelines.md` | なるべく作る |
| 出力のひな形 | `templates/{コマンド名}/template.md` | なるべく作る |
| 過去の事例 | `knowledge/{分野}/examples/` | あれば追加 |

### ステップ 3：README・一覧表を更新する

以下の 3 ファイルを**必ず**更新すること：

1. **`README.md`（プロジェクトルート）** — 「できること一覧」に新しいコマンドを追記する。非エンジニアが最初に見る場所なので、専門用語を使わず「何ができるか」「どう使うか」を簡潔に書く
2. **`.claude/commands/README.md`** — コマンド一覧に追記
3. **`knowledge/README.md`** — 追加したナレッジがあれば追記

### ステップ 4：成果物の出力ルール

- 成果物は `outputs/{分野}/YYYY-MM-DD_{案件名}/` に保存する
- 保存後に PR（レビュー依頼）を作成し、チームに確認してもらう
- PR のタイトルは `[分野] 案件名 - 成果物の種類` とする
  - 例：`[estimation] ○○案件 - 見積もり報告書`

## 分野の分類

コマンドやナレッジは以下の分野で整理する。必要に応じて追加OK。

| 分野 | 内容 | コマンド例 |
|------|------|-----------|
| `estimation` | 見積もり | `/estimate` |
| `sales` | 営業・提案 | `/proposal`（提案書作成）|
| `pm` | プロジェクト管理 | `/meeting-notes`（議事録）|
| `engineering` | 技術・開発 | `/code-review` |
| `hr` | 人事・採用 | `/interview-prep` |
| `general` | 汎用 | `/summarize`（要約）|

## コミットのルール

- コミットメッセージは**日本語**で書く
- ユーザーに確認せず、変更内容から適切なメッセージを自動で作成する
- 形式は以下の通り：
  - 1行目：変更の要約（何をしたか）
  - 2行目：空行
  - 3行目以降：補足（なぜ・どう変えたか）。不要なら省略OK
- **プッシュは自動で行わない**。プッシュはユーザーが自分で行う
- 例：
  ```
  見積もりコマンドを追加

  スプレッドシートから工数を自動算出し、結果を書き戻す機能を実装
  ```

## ファイルの書き方ルール

- ドキュメントは Markdown（`.md`）で書く
- ファイル名は英語の小文字でハイフン区切り（例：`meeting-notes.md`）
- 内容は日本語で書く
- パスワードや API キーなどの機密情報は `.env` に書き、Git には載せない
